# Spec-file authoring guidelines (consulted during Block 2)

## Spec file — general rules

- ASCII only unless non-ASCII is genuinely needed; save as UTF-8 if so.
- Every spec must carry the standard openSUSE copyright header. If missing, run `osc service run format_spec_file` to add one rather than hand-writing it.
- Use macros (`%{_bindir}`, `%{_libdir}`, `%{_mandir}`, `%{buildroot}`, `%{name}`, `%{version}`, …) instead of hard-coded paths. List at https://en.opensuse.org/openSUSE:Packaging_Conventions_RPM_Macros.
- Do not use the macro form of shell utilities (`%__cat`, `%__rm`, …); plain `cat`, `rm`, etc. are preferred. The spec-cleaner bot converts these automatically.
- Do not mix RPM macros and the auto-exported shell vars (`$RPM_BUILD_ROOT`, `$RPM_OPT_FLAGS`, …). Pick one style per spec; macros are the modern preference.
- Forbidden tags: `Copyright` (use `License`), `Packager`, `Vendor`. `AutoReqProv` only when you genuinely need to disable auto-deps.
- `License:` must be the SPDX short name (https://spdx.org/licenses/). Aggregate licenses with `and` / `or` and parentheses. Always declare the license on every subpackage too — don't rely on inheritance.
- **`Group:` — always drop.** Modern openSUSE Factory no longer requires the `Group:` tag, and the project policy in this skill is to remove it on sight. Do not add it to new specs and remove it from existing ones (`spec-cleaner --remove-groups` does this automatically; you can also delete the line by hand). Apply this in every spec edit even when the user didn't explicitly ask for it. The historical list at https://en.opensuse.org/openSUSE:Package_group_guidelines is only relevant if you encounter a legacy package that for some reason must keep the tag.
- `Source:` should be a full upstream HTTP(S) URL when possible.
- `Summary:` is one short line; package-description rules at https://en.opensuse.org/openSUSE:Package_description_guidelines.
- No `BuildRoot:` line and no `%clean` section in modern specs — they are obsolete since rpm 4.8.
- No pre-built binaries or libraries in the source package. Everything must build from source. Firmware and a few toolchain bootstraps are the documented exceptions.
- `gcc` and `make` are **not** pulled in by the default buildroot — any spec that compiles C/C++ or runs `make` must declare them explicitly (`BuildRequires: gcc`, `BuildRequires: make`). Specs inherited from older Factory revisions often omit these because the buildroot used to include them; add them when touching such a spec.
- **HARD RULE: when upstream states a *minimum version* for a dependency, encode it in the `BuildRequires` as `>= X.Y`** — don't leave a known floor unversioned. Read the minimums from upstream's own declarations and mirror them exactly: CMake `find_package(Foo X.Y REQUIRED)` / `pkg_check_modules(... foo>=X.Y)`, autotools `PKG_CHECK_MODULES`/`AC_*`, `doc/dependencies.md`, `pyproject.toml`/`setup.py`, `Cargo.toml`, etc. → `BuildRequires:  pkgconfig(libevent) >= 2.1.8`, `BuildRequires:  libboost_headers-devel >= 1.74.0`. The floor is real information (won't build/work against older libs), makes the constraint explicit for the resolver, and matters for backports to older distros. Only encode versions upstream actually states — don't invent a floor. (Real case: bitcoin 31.0's CMake declares Boost 1.74.0 / SQLite3 3.7.17 / Libevent 2.1.8 / Qt 6.2 minimums — all mirrored into the spec.)
  - **And the converse — on every version bump, RE-VERIFY the existing `>= X.Y` floors against upstream's *current* declared minimums; they drift.** A floor that was right three releases ago may now be (a) **too low** — upstream raised the requirement, so bump the spec to match (otherwise the constraint is a lie and backports can pick a too-old lib); (b) **stale/unnecessary** — upstream dropped or lowered it, so the floor is just historical noise (leave it if harmless and satisfied, but know it no longer reflects a real requirement); or (c) **newly stated** — upstream added a minimum for a dep your spec lists unversioned, so add it. Pull the current minimums from the new sources (CMake `find_package`/`gdal_check_package(... VERSION …)`/`VERSION_LESS` fatal gates, `pkg_check_modules`, `doc/dependencies*`, etc.) and diff against the spec's versioned `BuildRequires`. Keep any **intentional downstream floor** that exceeds upstream's (e.g. one forced by the distro toolchain) — recognise it by its explanatory comment and don't lower it. (Real case: gdal 3.13.0 — `proj >= 6.3`/`hdf5 >= 1.10`/`libtiff-4 >= 4.1`/`spatialite >= 4.1.2` still matched upstream; `OpenEXR >= 3` was an intentional C++17 downstream floor (kept); upstream's freshly-stated `CURL >= 7.68` was missing from the spec and was added.)
  - **A dependency can also change *kind* between releases — required ↔ optional, conditional → unconditional — not just its version floor; re-read the whole manifest, don't just diff the floors.** Watch for: (a) a previously-**conditional** dep becoming **unconditional** (e.g. a Python `typing-extensions` guarded by `%if python_version < 311` that upstream now lists in plain `dependencies` at `>= 4.5` → drop the `%if`, make it a flat `BuildRequires`/`Requires >= 4.5`); (b) a previously-**required** dep becoming **optional** (moved into `[project.optional-dependencies]` / a Cargo `[features]` / a CMake `option()`) → drop the hard `Requires`, and if the optional feature is worth keeping enabled by default, express it as `Recommends:` (installed by default on openSUSE) rather than `Requires:` — but first confirm the test suite doesn't need it (grep the test dir for the import); (c) a brand-**new** mandatory dep (see "Submitting a brand-new package" for the Factory-availability blocker). (Real case: python-pytools 2024.1.5→2026.1.1 — `typing-extensions` went unconditional `>= 4.5`, `numpy` moved to an optional extra (→ `Recommends:`, tests don't import it), and `siphash24 >= 1.6` was newly required.)
- Do not bundle multiple upstream projects in one package; do not duplicate system libraries (patch to use the system copy instead).

## `%prep` / `%build` / `%install` / `%check`

- Prefer `%autosetup -p1` over manual `%setup` + `%patch` sequences (uses git internally for clean patch tracking).
- If you must use `%setup`, pass `-q` to keep the log small.
- **Build C/C++/Fortran with the distribution flags (`%{optflags}` + the distro `LDFLAGS`) — for binaries *and* tests — unless there is a very good, documented reason not to.** This is mandatory policy, not a nicety: the distro flags carry optimization, `_FORTIFY_SOURCE`, stack-protector, PIE/relro and other hardening. You may **extend** them with extra flags the package needs, but do not **replace** or drop them.
  - If upstream's build system *hardcodes* its own flags (e.g. a Makefile with `CFLAGS = -g -Wall -Werror` that overrides the environment, or link lines without `$(LDFLAGS)`), patch it to honour the passed-in `CFLAGS`/`LDFLAGS` rather than leaving the distro flags ignored. (Real case: the packetdrill SUSE patch exists precisely to make its Makefile respect `$(CFLAGS)`/`$(LDFLAGS)`.)
  - **Verify the flags are actually honoured** — don't assume. Read the build log's compile/link lines and confirm `%{optflags}` (e.g. `-O2 -fstack-protector-strong -D_FORTIFY_SOURCE` …) and the link hardening flags really appear; a build that *succeeds* while silently ignoring the flags has not done its job. `grep` a representative compile line in the `osc build` output.
  - If honouring the flags **breaks** the build or tests (e.g. `-O2`/LTO exposes latent upstream UB), the fix is to address the root cause or add a *minimal targeted* extra flag (e.g. `-fno-strict-aliasing`, or as a last resort dropping `-flto`) — **with an in-spec comment explaining why** — not to abandon the distro flags wholesale. Only fully stripping/filtering flags needs a strong documented justification.
  - **The rule is "don't *drop* the distro flags," not "inject every flag everywhere."** `%{optflags}` (CFLAGS — carries `-O2` and hardening) is the core requirement and belongs on every C/C++/Fortran compile. But do **not** reflexively add `%{?build_ldflags}` to a hand-written `gcc` one-liner that never used it, and don't bundle such an unrequested change when you were only asked a narrow question. Adding link-hardening (`-pie`/relro/now) to a trivial single-file tool is often pointless churn, and for a **benchmark/performance** binary it can be actively unwanted (it perturbs the very thing being measured). Treat LDFLAGS injection into a manual compile line as a per-package judgement call — when in doubt leave it, or ask. (Real case: `ebizzy` — adding `%{?build_ldflags}` was rejected as useless for a benchmark.)
- **Strive to build *verbosely* so the actual compile/link command lines are visible — you can't verify the distro flags are honoured if the build hides them.** A silent build that prints only `CC foo.o` gives you nothing to check `%{optflags}`/LDFLAGS against. Make the build echo full commands: **`%make_build` already injects `V=1 VERBOSE=1`** (so a make-based build through it is already verbose — do *not* add an explicit `V=1`, spec-cleaner strips the redundant one); `%cmake` already sets `CMAKE_VERBOSE_MAKEFILE:BOOL=ON` (so plain `%cmake_build` is verbose; with Ninja add `-v` if needed); only a *raw* `make` call (not via the macro) needs `make V=1` / `VERBOSE=1` added by hand; meson/`%meson_build` is verbose by default. Then `grep` a representative compile line in the log and confirm `-O2 -fstack-protector-strong -D_FORTIFY_SOURCE` … and the link-hardening flags really appear (see the distro-flags rule above). **Treat verbose-build + distro-flags as the default to aim for, with documented exceptions:** a build system that genuinely can't be made verbose, a case where the flags legitimately can't be applied (and is commented as such), or a language that manages its own flags (Go/Rust/Cargo, Python wheels, Java) where `%{optflags}` doesn't map directly — there, verify via the toolchain's own mechanism instead. When you *can* be verbose and use the flags, do.
- Strip upstream `-Werror` (often via `--disable-werror` or a small patch) — new toolchains turn fresh warnings into errors and break builds.
- Use `%make_build` and `%make_install`. Do not use the legacy `%makeinstall` (no underscore). For non-parallel builds, write `make -j1` and add a comment explaining why.
- **CMake packages: use `%cmake` / `%cmake_build` / `%cmake_install` — not `%make_build`/`make` — always.** The macros (from the `cmake` package's `cmake.macros`) drive the build generator-agnostically (`%cmake_build` runs `%__builder`). **For a time-consuming CMake build, switch the generator to Ninja:** add `BuildRequires: ninja` and **`%define __builder ninja` immediately before the `%cmake` call** (placement matters — `%cmake` reads `%__builder` to decide the generator: anything other than make makes it pass `-GNinja`; `%cmake_build`/`%cmake_install` then run `ninja`). Ninja parallelizes large multi-target builds (e.g. multi-plugin Qt apps) noticeably better than make. Confirm it took effect by checking the configure line shows `-GNinja` and the build log shows `ninja: Entering directory`. **spec-cleaner does NOT rewrite the legacy build-step macros** — a CMake spec using `%make_jobs` or `%make_build` (or a bare `make %{?_smp_mflags}`) for the build step passes spec-cleaner unchanged, so fix it by hand to `%cmake_build` (`%make_jobs` is deprecated and the build log warns about it). Real case: diskscan used `%cmake` + `%make_jobs` → switched to `%cmake_build`.
- **HARD RULE: any `-j1` / `--jobs 1` / `%make_build -j1` (serial-build) constraint is something you MUST actively test during the local build pass — never carry it forward unexamined.** On every package you build locally (not just major bumps), if the spec forces a serial build, drop the `-j1` and rebuild `--clean` to find out whether parallelism is actually safe *now*. There are exactly two legitimate outcomes, and you must reach one of them:
  - **Parallel works → remove the `-j1` and its comment** (and, if upstream's CI splits targets, mirror that split per the next bullet). This is usually a big wall-clock win. Real case: fwts kept `# parallel build fails on Factory` + `%make_build --jobs=1` long after upstream fixed the underlying race; removing it cut build time from ~92 min to ~25 min on Tumbleweed/x86_64. When parallel is restored, mention the validation in `.changes` so reviewers can trust the change.
  - **Parallel genuinely still races → keep `-j1` and leave a precise in-spec comment recording the *root cause* you found** (which target/objects race, and why), so the next maintainer doesn't re-litigate it. Reverting to the existing `-j1` is a no-net-change round-trip, so it gets **no `.changes` bullet** — the spec comment is the entire artefact. Real case: **xmlrpc-c** — its recursive makefiles miss a dependency between the per-source `.osh` shared objects and the `.so` link target, so at high `-j` the shared-library link launches before its `.osh` inputs finish; with the distro's `-flto=auto` this surfaces as a fatal `lto1: internal compiler error: resolution sub id … not in object file`. Serial build passes, parallel fails → `--jobs 1` is load-bearing and was kept with a comment. (Disabling LTO would not fix it — the missing-dependency race is independent of LTO, which only makes the corruption louder.)
  The point of the hard rule is that "it has `-j1`" is never a settled fact you inherit — it is a hypothesis you re-test locally and then either kill or document with evidence.
- **If a parallel build still races after the re-test, check how upstream CI builds before falling back to `-j1`.** Many autotools projects mix multiple top-level targets (`make all training doc`, `make all check`, …) in a *single* `%make_build` line, and the libtool/automake convenience-library link rules race when the same `.lo` files are needed by sibling targets. Look at the project's GitHub Actions / CI workflow (`.github/workflows/*.yml`) — if upstream invokes the targets across **multiple** `make` invocations in sequence (e.g. `make -jN` then `make training -jN`), mirror that in the spec instead of collapsing to `-j1`. Two parallel `make` calls beat one serial call by a wide margin. Real case: tesseract-ocr 5.5.2 races at `libtesseract_ccutil.la` when `%make_build -jN all training doc` is used together; splitting to `%make_build all` + `%make_build training doc` (as upstream's autotools.yml does) builds cleanly in parallel. Document the split with a short in-spec comment so a future maintainer doesn't recombine them. Worth a brief `.changes` bullet whenever the change demonstrably restores parallel build (developer-visible wall-clock win); silent in `.changes` if the split has no measurable speedup.
- Never `rm -rf %{buildroot}` at the top of `%install` — it opens a tmp-race; rpm cleans the buildroot for you.
- `%install` must not `chown` (rpmbuild runs unprivileged). File ownership belongs in `%files`.
- Scriptlets (`%pre*`, `%post*`) may only touch `%buildroot`, `%_builddir`, and `/tmp` / `/var/tmp` / `%_tmppath`. Anything used by a scriptlet must appear in `Requires(pre):` / `Requires(post):` etc.
- **GENERAL RULE — always try to ship a `%check`.** Every package, new or updated, should run upstream's test suite during the build whenever it can possibly be made to run. rpmlint's `no-%check-section` warning is a prompt, not a pass: treat a package with no `%check` as incomplete until you've confirmed upstream genuinely has no tests. Before settling for none, look in the source tree for a `test/`/`tests/`/`spec/` dir, a `ctest`/`CTEST`/`-DBUILD_TESTING` knob, a `pytest`/`tox.ini`/`pyproject` test config, a `cargo test`, a `Makefile` `test`/`check` target, or a self-contained test program. Wire whichever applies (`%ctest`, `%pytest`, `cargo test`, or compiling + running the test binary). **This applies even to header-only / "nothing to compile" libraries** — they routinely ship a standalone test program you can compile against the installed headers and run: e.g. bshoshany-thread-pool ships `tests/BS_thread_pool_test.cpp` (no framework, its own `main`, deadlock/benchmark suites off by default) → `g++ %{optflags} -std=c++17 -pthread -Iinclude tests/...cpp -o t && ./t tests` turns a `no-%check` warning into 172 gating checks (add a `BuildRequires: gcc-c++` for the compiler; the package stays `noarch`). A `%check` that actually runs is one of the highest-signal quality wins for a Factory submission.
- **HARD RULE: re-evaluate a disabled/commented-out `%check` (or any disabled test) on *every* touch of the package — never assume "tests are disabled" is still true.** A `%check` commented out with a stale "tests fail" note is a hypothesis to re-test, exactly like a `-j1` workaround (above) or a `||:` masker. Try enabling it and rebuild; a new upstream release very often fixes whatever broke the tests. The single most common reason a suite "fails" in the offline OBS build is that it **tries to download test data/fixtures at test time** — look for a knob to point at local data or to disable just the data-dependent tests, so the rest of the suite still runs and gates the build. (Real case: libzim's `%check` was fully commented with "test mostly fail now"; the cause was meson's default `test_data_dir=''` downloading a test-data suite — building with `-Dtest_data_dir=none` compiles the suite in no-external-data mode and **all 31 tests pass** offline.) Likewise re-check `--disable-tests`/`-DBUILD_TESTING=OFF`/`%bcond`-gated tests. If after a genuine attempt the tests still can't run offline (need root, network, special hardware), keep them off **with a precise comment stating the real reason** — not a vague "tests fail" — and prefer running at least the subset that does work. Re-enabling a previously-disabled suite warrants a brief `.changes` bullet.
- **For CMake/CTest test suites, run them with `%ctest`** (it `cd`s into `%__builddir`, runs `ctest --output-on-failure`, and surfaces per-test results). Do **not** hand-roll `cd %{__builddir}; %cmake_build <target>; ./tests/foo` — `%cmake_build` does *not* `cd`; it relies on the cwd already being the build dir, which is true inside `%build` (because `%cmake` cd's in and the whole section is one shell) but **not** in the fresh `%check` shell.
- **`%ctest` finding zero tests is a silent green, not a pass.** A suite that prints `No tests were found!!!` exits 0, so `%check` succeeds while testing nothing. Always read the `%check` log and confirm a real `N/N Test #… Passed` / `100% tests passed … out of N` summary. Common reasons ctest finds nothing, all needing a small patch:
  - **No `add_test()`** — upstream calls `enable_testing()` but never registers the binary. Add `add_test(NAME foo COMMAND foo)` (or `gtest_discover_tests(foo)`).
  - **`enable_testing()` only in a subdirectory.** `%ctest` runs from the *top* build dir; if `enable_testing()` is called only inside `tests/CMakeLists.txt`, the top-level CTest file never recurses into it. Add `enable_testing()` to the **top-level** `CMakeLists.txt` (before `add_subdirectory(tests)`).
  - **Test binary is `EXCLUDE_FROM_ALL`.** `%ctest` only *runs*, it never *builds*; an excluded target is never built by `%cmake_build`, so ctest has no binary to invoke. Drop `EXCLUDE_FROM_ALL` so it builds with everything.
- **Don't let a test suite download its test framework at configure time.** Many CMake projects `git clone`/`FetchContent` googletest (or Catch2, …) during configure — that needs network and **fails in the offline OBS build**. Patch it to use the system copy: `find_package(GTest REQUIRED)`, link `GTest::gtest_main GTest::gmock`, and `BuildRequires: pkgconfig(gtest)` + `pkgconfig(gmock)` (the same package that ships the headers/libs also provides the CMake config `find_package(GTest)` needs).

## `%files`

- **Ship shell completions in their own `noarch` subpackage, one per shell — don't bundle them in the main package.** A completion script is arch-independent data that a user only benefits from if the matching shell is installed, so split it out with a `Supplements:` so it auto-installs alongside that shell. Pattern (drop `Group:` per project policy; the template some specs use still shows it):
  ```
  %package bash-completion
  Summary:        Bash completion for %{name}
  Requires:       %{name} = %{version}
  Requires:       bash-completion
  Supplements:    (%{name} and bash-completion)
  BuildArch:      noarch

  %description bash-completion
  Bash command-line completion support for %{name}.

  %files bash-completion
  %{_datadir}/bash-completion/completions/<file>
  ```
  Analogous `%package fish-completion` (`Supplements: (%{name} and fish)`, files under `%{_datadir}/fish/vendor_completions.d/`) and `%package zsh-completion` (`Supplements: (%{name} and zsh)`, files under `%{_datadir}/zsh/site-functions/`). The bash one additionally `Requires: bash-completion` (it owns the completions dir); fish/zsh don't need a dir-owner requires. Moving an existing in-main completion into such a subpackage is consumer-visible (the file moves packages) → give it a `.changes` bullet. (Real case: partclone — bash completion split into `partclone-bash-completion`.)
- Own every file you install; never own files that belong to another package. Own intermediate directories where appropriate (see Specfile guidelines for the perl/multi-package directory rules).
- No duplicate file entries across the main package and subpackages. Run `%fdupes %buildroot` (with `BuildRequires: fdupes`) to hardlink identical content.
- **`%fdupes` runs in `%install`, but `%doc <file>` in `%files` copies docs *after* `%install` finishes** — so `%fdupes %{buildroot}%{_docdir}/%{name}` errors with "could not chdir" when there are only `%doc`-staged files. To dedupe upstream's doc set (e.g. an identical `NEWS` and `ChangeLog`): install them manually in `%install` (`install -d %{buildroot}%{_docdir}/%{name}` + `cp -p AUTHORS ChangeLog NEWS README %{buildroot}%{_docdir}/%{name}/`), then `%fdupes` that dir, then list it as `%doc %{_docdir}/%{name}` in `%files` instead of listing individual files.
- Mark license files with `%license`, not `%doc`. For backward compatibility with pre-Leap-15 targets, gate on `%if 0%{?suse_version} < 1500` and fall back to `%doc`.
- Mark config files `%config(noreplace)` unless overwriting the user's edits is genuinely correct; comment when using plain `%config`. Never use `%config` under `/usr`.
- Use `%find_lang` and `%lang_package` for translations. Do not glob `%_datadir/*` to scoop locale files (it claims directory ownership you don't own).
- Keep development artefacts (`*.h`, unversioned `lib*.so`, `*.pc`, static libs) in a `-devel` subpackage. Packages shipping `.pc` files need `BuildRequires: pkgconfig`.
- **A subpackage that ships only arch-independent content (config files, data, scripts — no compiled binaries) should be `BuildArch: noarch`**, otherwise rpmlint flags `E: no-binary` (an arch package with nothing arch-specific in it). Add `BuildArch: noarch` to that subpackage's preamble (spec-cleaner orders it *after* the dependency tags). It coexists fine with arch siblings that `Requires:` it `= %{version}`. (Real case: libirman's config-only `irman-common` subpackage → noarch cleared the `no-binary` error.)
  - **A header-only `-devel` subpackage (only `*.h`, `.so` lives in the main/lib package) also trips `no-binary` and is fixable with `BuildArch: noarch` — but VERIFY the headers are truly arch-independent first.** Before adding `noarch`: (1) confirm the header is installed **verbatim**, not generated/preprocessed at build time — check the build for `configure_file(...)`, a `*.h.in` template, or `sed`/`autoheader` munging of the installed header (a configure-generated header can bake in arch/word-size/endian values and is therefore arch-specific); and (2) grep the header for arch-conditional content (`__x86_64__`, `__aarch64__`, `sizeof`-derived constants, endian macros). Only when both checks are clean is `noarch` safe. (Real case: lzfse — `install(FILES src/lzfse.h)`, no `configure_file`, no arch macros → noarch devel cleared `no-binary`.)
- **`W: shared-dir-glob-in-files` — don't glob a shared system dir in `%files`.** A line like `%{_includedir}/*` or `%{_datadir}/cmake/*` claims everything under a directory owned by many packages. List the package's actual entry instead: the specific header (`%{_includedir}/glpk.h`) or the package's own subdir (`%{_includedir}/unrar/`, which owns the dir and its contents). (Real cases: glpk, unrar.)
- **`E: obsolete-suse-version-check NNNN` — drop the dead version guard.** rpmlint flags `%if 0%{?suse_version} == 1310` (or other no-longer-built values) as referencing a `suse_version` no current target uses. If the devel project builds nothing that old (`osc repos <prj>`), the whole `%if … %endif` is dead code — remove it. (1310 = openSUSE 13.1 / SLE12-era.) Distinguish from a *live* version gate that still protects a built old distro (see the patch caveat above) — that one stays. (Real case: bogofilter — two dead `== 1310` Conflicts blocks removed.)
- **Leap 15.6 is EOL (as of 2026-06) — treat the lowest openSUSE target as 16.0 (`suse_version` 1600), and proactively remove `%if 0%{?suse_version} < 1600 … %endif` blocks.** These guarded Leap 15.x / SLE15 and are now dead code: keep the `%else` (>= 1600) branch and drop the rest. The most common cases are **old-compiler fallbacks** (`gcc12`/`gcc13`/`gcc13-c++` BuildRequires and `export CC=gcc-13`/`-DCMAKE_C_COMPILER=gcc-12` in `%build` — 16.0+ has a new-enough default `gcc`), and version-gated `%bcond` toggles (collapse `%if … >= 1600 %bcond_with foo %else %bcond_without foo %endif` to just the 16.0 behavior, e.g. plain `%bcond_with foo`). A 15.6 *build failure* in a devel project is not a Factory blocker — judge server-side green on Tumbleweed/16.0 and treat a lingering 15.6 repo as informational. (Real cases: gpsbabel — dead `gcc12` fallback for `< 1550`; proj — `gcc13-c++`/`gcc-13` fallback for `< 1600` removed once 15.6 went EOL.) Note the threshold differs by package: `< 1550`, `< 1600`, etc. — remove whichever guards only EOL distros; a guard still protecting a *built, non-EOL* target stays (the patch/version-gate caveat above).
- **rpmlint *errors* don't fail the build unless total badness exceeds the project threshold (999).** Some `E:` lines are inherent to a package's design and are accepted in Factory: `update-alternatives`-based packages routinely emit `dangling-symlink` (the `/etc/alternatives/*` target is created in `%post`) and `alternative-generic-name-missing`; multibuild source packages emit `invalid-spec-name` for the `<pkg>-<flavor>-src` artifact; migration stanzas emit `self-obsoletion`. Don't churn the alternatives/multibuild machinery to silence these — confirm they're **pre-existing** (present before your edit, and the package builds in Factory with them) and leave them. Focus your rpmlint fixes on errors your change introduced or genuine dead code. (Real case: bogofilter carried 8 such pre-existing errors at badness 8 ≪ 999.)
- **A documentation subpackage (`-doc` / `-devel-doc`) must not `Requires:` the `-devel` (or any other) subpackage.** Generated API docs (Doxygen, etc.) are useful standalone and should be installable without dragging in headers. The hard dependency also trips rpmlint's `devel-dependency` error (a non-`-devel`-named package depending on a `-devel` package). Drop the `Requires:` line; do not add a weaker `Recommends:` unless there's a concrete reason. This is consumer-visible (the produced RPM's deps change), so it warrants a `.changes` bullet.
- **Self-referential `Provides`/`Obsoletes` for a subpackage that no longer exists are usually droppable cruft.** A pair like `Provides: %{name}-doc >= %{version}` + `Obsoletes: %{name}-doc < %{version}` is the standard upgrade-path stanza left behind when a former `-doc` (or other) subpackage was *merged back* into the main package — it lets an installed `<name>-doc` be replaced cleanly on upgrade. But if that subpackage **never actually shipped in a released distro** (it was added speculatively, or the split only ever lived in the devel project), the stanza guards an upgrade path that can't occur and is just noise — drop both lines. **Caveat before removing:** confirm the subpackage really never shipped — if `<name>-doc` (or whatever) *was* in a released openSUSE/Leap, the `Obsoletes`/`Provides` are load-bearing (they carry the upgrade) and must stay. Removing them changes the RPM's `Provides`/`Obsoletes` metadata, so it's consumer-visible → give it its own `.changes` bullet. (Real case: `asl` carried `Provides/Obsoletes %{name}-doc` with no `asl-doc` subpackage anywhere; dropped.)
- Filenames containing non-ASCII characters must be encoded UTF-8 — convert with `convmv` in `%install` if upstream ships another encoding.

## Shared libraries

https://en.opensuse.org/openSUSE:Shared_library_packaging_policy

- A versioned shared library `libfoo.so.N[.x.y]` belongs in its own subpackage named `libfoo<N>` (or `libfoo-<VER>-<SONUM>` for the full form) with the SO version and dots-to-underscores encoded into the package name. Use `%package -n libfoo<N>` to set the name explicitly.
- The versioned shlib subpackage contains *only* the `.so.*` files (plus optionally the license in a versioned dir). Headers, `.so` symlinks, `.pc` files, `.la` files go in `-devel`.
- Avoid packaging static libraries; use `--disable-static`. Drop `.la` files unless the library lives outside `%_libdir`. **A version bump can *newly* start installing a static `lib*.a` that a prior release didn't** — it then surfaces as `error: Installed (but unpackaged) file(s) found: …/lib*.a` at the packaging stage (the compile/install succeed; rpm fails at the end). Fix by disabling the static build if CMake/configure offers a toggle, else `rm -f %{buildroot}%{_libdir}/lib*.a` in `%install`. (Real case: libcec 7.0.0→7.1.1 began installing `libcec.a`.) More generally, after any version bump re-check `%files` against what actually gets installed — bumps add/move/rename files (new static libs, relocated python bindings, new manpages) and the mismatch only shows up as unpackaged-files or a missing-file error at the very end of the build.
- If upstream ships an unversioned `libfoo.so`, patch the build to add a SONAME (e.g. `libfoo.so.suse0`) rather than packaging an unversioned shared library.
- **On a version bump of a shared library, verify the SONAME/soversion from the *upstream source*, not by guessing — then bump the spec's `%define sover...` to match.** Don't assume "minor bump → soname unchanged" or "+1". Read the authoritative value from the build system: CMake `set(GDAL_SOVERSION 39)` / `SOVERSION` target property, autotools `-version-info`/`libtool` current:revision:age, or the produced `lib*.so.N`. (Real case: gdal 3.12→3.13 bumped `GDAL_SOVERSION` 38→39 in `gdal.cmake`; the spec's `%define soversion` was updated to 39 and the `libgdal38`→`libgdal39` subpackage rename followed.) A soname change then triggers the lib-subpackage rename **and** the `baselibs.conf` update below, and Factory staging will rebuild the library's consumer cone (expect a longer staging dwell).
- **HARD RULE — on a soname/lib-subpackage rename, also update `baselibs.conf`.** When a version/ABI bump renames the shlib subpackage (`libfoo1`→`libfoo2`, or full-version names like `libunrar7_2_5`→`libunrar7_2_6`), grep the package's `baselibs.conf` (often `Source99`) for the old library package name and update *every* occurrence to the new `%package -n` name. `baselibs.conf` hardcodes those names verbatim to generate the `-32bit`/`-64bit` baselibs packages; spec-cleaner and your spec edits don't touch it, so it silently goes stale and produces wrong/broken baselibs. (Real case: unrar 7.2.5→7.2.6 — `baselibs.conf` still said `libunrar7_2_5`.)
- **A major version that splits/renames a library can ship a *backward-compat* copy whose real SONAME differs from its filename — package it in the subpackage whose name matches the SONAME, or rpmlint's `shlib-policy-name-error` (badness 10000) aborts the build.** When upstream moves the implementation into a new library but keeps installing the old `lib<old>.so.N` name for source/link compatibility, that compat file often carries the *new* library's `DT_SONAME`. rpmlint's shared-library policy keys the **package name suffix off the SONAME version**, not the filename — so shipping `lib<old>.so.<N>` (SONAME `lib<new>.so.<M>`) in a `lib<old><N>` subpackage trips `shlib-policy-name-error … expected package suffix: <M>`. Fix by putting the compat file in the **`lib<new><M>`** subpackage (suffix matches the SONAME); its `.pc`/`-devel` symlink redirect to the new lib anyway. Check the real SONAME with `objdump -p <file> | grep SONAME` before assigning subpackages. Also: upstream may install that compat lib via `install(FILES …)` (non-executable, 0644 → rpmlint `shared-library-not-executable`) and create its symlinks via an `install(CODE …)` rule that ignores `DESTDIR` (so they never reach the buildroot) — fix the bit with a patch (`install(PROGRAMS …)`, upstreamable) and create the symlinks yourself in `%install`. (Real case: mbedtls 4.1.0 — crypto moved to TF-PSA-Crypto; `libmbedcrypto.so.4.1.0` has SONAME `libtfpsacrypto.so.2`, so the compat libs went into `libtfpsacrypto2`, not a `libmbedcrypto18` subpackage; `libeverest`/`libp256m` subpackages disappeared entirely.)

## Patches

https://en.opensuse.org/openSUSE:Packaging_Patches_guidelines

- Every patch needs a comment line directly above its `PatchN:` tag tagging its category, e.g. `# PATCH-FIX-UPSTREAM name.patch bsc#NNNNNN you@example.com -- short reason`.
- Categories: `FIX` / `FEATURE` × `UPSTREAM` / `OPENSUSE` / `SLE`, plus `NEEDS-REBASE` for temporarily-disabled patches.
- Upstream-eligible patches *must* be sent upstream — note the upstream PR/commit URL in the tag.
  - **One independent issue → one patch → one upstream PR. When a single build break has *multiple independent root causes*, split them into separate patches and open a separate PR for each — never bundle them.** Different root causes have different reviewers, different merge timing, and different revert risk upstream; a combined patch/PR is harder to review and can be blocked wholesale by an objection to just one part. Tell them apart by asking "could one fix land without the other?" — if yes, they're independent. Each patch gets its own `PatchN:` + `# PATCH-FIX-UPSTREAM … <repo>#<PR>` tag. (Real case: soapy-uhd FTBFS had two unrelated causes — UHD 4.x headers needing **C++17** (`cxx17.patch` → SoapyUHD#75) and a missing **`<boost/lexical_cast.hpp>`** include after Boost dropped the transitive one (`boost-lexical-cast.patch` → SoapyUHD#76) — shipped as two patches + two PRs.)
  - **Actually file the upstream PR/MR when the fix is a genuine upstream bug (e.g. a build-system defect every packager hits), then reference it.** Keep the patch **minimal and faithful** to the one real bug (don't bundle unrelated cleanups), and **verify the bug still exists on the upstream default/development branch** before opening it — sometimes part of your fix is already merged post-release (`gh api repos/<o>/<r>/contents/<path>?ref=<default-branch> --jq .content | base64 -d`), so narrow the patch to only what's still broken. Mechanics via the GitHub API (works without a local clone of a huge repo): `gh repo fork <o>/<r> --clone=false`; `gh repo sync <you>/<r> --branch <default>`; create a branch ref (`gh api -X POST repos/<you>/<r>/git/refs -f ref=refs/heads/<branch> -f sha=<HEAD>`); commit the edited file via `gh api -X PUT repos/<you>/<r>/contents/<path> -f message=… -f content=<base64> -f sha=<filesha> -f branch=<branch>`; `gh pr create --repo <o>/<r> --base <default> --head <you>:<branch>`. **DCO gotcha: the `Signed-off-by:` email must match the commit *author* email** (the API sets the author from your GitHub account's git email) — a mismatch fails the project's DCO check. If they differ, reset the branch ref to HEAD and re-commit with a matching sign-off. Then add the patch downstream as `# PATCH-FIX-UPSTREAM <file> gh#<o>/<r>#<PR> -- <reason>`. (Real case: Mbed-TLS/mbedtls#10777 — `install(FILES …)` → `install(PROGRAMS …)` so the libmbedcrypto compat shared object is installed executable; the DESTDIR half of the original bug was already fixed on `development`, so the MR was narrowed to just the executable-bit fix.)
  - **Non-GitHub upstreams: use the matching forge CLI/API.** **Codeberg/any Gitea** → `tea` (the same client used for `src.opensuse.org`): check `tea logins list`; if the upstream forge isn't there, the login needs a token *the user* generates (`write:repository`+`write:issue` scopes) — have them run `! tea logins add --name codeberg --url https://codeberg.org --token <TOKEN>` so the secret stays on their side. Then fork via the API (`curl -X POST -H "Authorization: token $TOK" .../repos/<o>/<r>/forks`), push the branch with the token via `GIT_ASKPASS` (a one-line script that `printf`s the token — **never** embed it in the remote URL or echo it), and open the PR via `POST .../repos/<o>/<r>/pulls` with `{"head":"<youruser>:<branch>","base":"<default>",…}`. **GitLab** → `glab`. Confirm your fork username from the API (it may differ from your OBS/codeberg display name — e.g. codeberg user `<your-forge-user>`, not `<your-obs-account>`). (Real case: dnkl/yambar#497 opened on codeberg via `tea`/API.)
- Apply patches with `%autopatch -p1` (preferred) or `%patch -P N -p1`. Do not delete a patch declaration when temporarily disabling — comment out only the `%patch` application and re-tag as `PATCH-NEEDS-REBASE`.
- **Use `%autosetup -p1` (with the explicit `-p1`), not a bare `%autosetup`, for git-style `a/…`→`b/…` patches.** A bare `%autosetup` failed to apply a standard `diff -u a/src/foo b/src/foo` patch with `No file to patch. Skipping patch. / 1 out of 1 hunk ignored` — the strip level didn't drop the `a/` prefix. Always pass `-p1` (it also matches how `git diff` / most upstream patches are generated). The skill already recommends `%autosetup -p1` in the `%prep` rules; this is the concrete failure mode when it's omitted.
  - **Root cause + a nastier variant: a bare `%autosetup` invokes `patch` with *no* `-p` flag, so GNU patch falls back to stripping down to the *basename*.** With a single-component prefix this just fails outright (above). But with a **multi-file patch** it silently *misroutes* hunks: a patch touching both `CMakeLists.txt` and `tests/CMakeLists.txt` had its `a/tests/CMakeLists.txt` hunk reduced to basename `CMakeLists.txt` and applied to the *top-level* `CMakeLists.txt`, where the context didn't match → `1 out of 1 hunk FAILED -- saving rejects to CMakeLists.txt.rej`. The patch applied fine with explicit `patch -p1` in a manual test, which is the tell that the strip level (not the patch) is wrong. Always `-p1`.
- **Prefer applying *all* patches unconditionally (`%autosetup -p1` / `%autopatch -p1`) over guarding individual `%patch` lines with `%if`/`%ifarch` — conditional patch application is discouraged.** When you see `%setup` + `%ifarch … %patch -P N … %endif` or `%if 0%{?suse_version} … %patch … %endif`, check whether the conditionality is actually needed and collapse it:
  - **Arch-gated `%patch` is almost always wrong** — the patch should be made portable so it applies everywhere. (Real cases: clFFT's "aarch64" patch merely *removed* bogus hardcoded `-m64`/`-m32` CMake flags — harmless on x86_64, so applied to all arches; itpp's C++-standard and `any()` fixes are compiler-portable.) After de-conditionalizing, build the *previously-excluded* arch too to confirm it's genuinely safe.
  - **Mixed `-p` levels are NOT a reason to keep `%if` guards** — they're a reason to keep an explicit sequence: `%autosetup -N` + `%patch -P 0` (its `-p0`) + `%autopatch -m 1 -p1` (the rest at `-p1`). No conditionals involved. (Real case: itpp.)
  - **CAVEAT — a `%if 0%{?suse_version} <op> NNNN %patch` guard can be load-bearing.** If the devel project still builds the old distro the guard protects, removing it will apply the patch there and may break it. Check `osc repos <prj>` for old targets and their `suse_version` before unconditionalizing: e.g. `server:mail` still builds `SLE_12_SP5` (`suse_version` 1315), so bogofilter's `%if 0%{?suse_version} > 1320 … %patch …` legitimately skips the patch there — **leave it**. Distro-version gating for genuinely-different distros is fine; only spurious arch/version guards on portable patches should be removed.
- Document every patch life-cycle event (add/rebase/modify/disable/drop) as a line in the `.changes` file. **A rebase counts** — when a version bump makes a patch stop applying and you regenerate it, add a brief `.changes` bullet saying so (e.g. "Rebase foo.patch for the new upstream sources"). Don't let a silently-rebased patch slip into a commit unmentioned.
- **HARD RULE: when you remove a patch (or any `Source`) file, name the *exact filename* in the `.changes` — `factory-auto` declines the SR otherwise, and the LOCAL `source_validator` does NOT catch this.** factory-auto's source-validator cross-checks every file *deleted* in the commit against the `.changes` diff and rejects with *"A patch (`<file>`) is being deleted without this removal being mentioned in the changelog."* An abbreviated/globbed mention is **not** enough — writing `- Drop the two backport-*.patch files` or `- Drop obsolete patches` fails the check; the literal `backport-050a8c2….patch` / `backport-b5321a88….patch` must each appear verbatim. The trap: `osc service runall source_validator` passes locally (it doesn't run this particular cross-check), so you only discover it as a server-side decline. So: for every `osc rm`/`git rm` of a patch or source, put its exact filename in a `.changes` bullet (one per file, or a list of literal names under a heading). Fix-and-resubmit a decline of this kind by amending the in-flight entry with the literal names and `osc sr … --supersede <declined-id>`. (Real case: nqptp 1.2.8 — `- Drop the two backport-*.patch files` was declined; spelling out both filenames and resubmitting fixed it.)
- **When a distro patch stops applying after a version bump, triage each hunk before rebasing — some may be obsolete because upstream adopted them.** Fetch the patched file at the new revision (`gh api repos/<o>/<r>/contents/<path>?ref=<rev> --jq .content | base64 -d`) and compare against what the patch intended. Hunks upstream has since incorporated (e.g. dropping a `-static` link flag) should be **dropped from the patch entirely**, not force-fitted; hunks still needed (e.g. respecting `$(CFLAGS)`/`$(LDFLAGS)` the Makefile still hardcodes) must be **regenerated against the new file**. Reconstruct cleanly with a diff tool: put the new upstream file in `a/`, edit a copy in `b/`, `diff -u a/<path> b/<path>`, and reassemble the patch with its original `From:`/`Subject:`/`Patch-mainline:` header preserved. Real case: packetdrill's 2018 SUSE Makefile patch — its `-static` removal was already upstream's default (dropped that hunk), while its CFLAGS/LDFLAGS-respecting hunks were rebased against the restructured `Makefile.common`.
- **Reproduce build/`%check` issues inside the build environment (`osc chroot`), not on your host — this is the preferred way of working.** The OBS build root has the exact toolchain and build deps (`flex`, `bison`, the right gcc, …) that your host may lack; debugging on the host gives misleading results and tempts you into polluting it with `zypper install`. `osc chroot` is `osc build --shell`: enter the *preserved* root from the last build with the **same** repo/arch/spec args plus `--noinit`, and run commands non-interactively with `--shell-cmd`:
  ```
  osc chroot --noinit --alternative-project <prj> <repo> <arch> <pkg>.spec --shell-cmd 'bash /tmp/repro.sh'
  ```
  (Drop a script into `<buildroot>/tmp/` via `sudo tee` first; the source tree lives under `/home/abuild/rpmbuild/BUILD/...` inside the root. Without explicit repo/arch matching the build, `osc chroot` may pick a non-native arch and error with *"native compile is not possible, an emulator … must be configured"*.) Use this to bisect compiler-flag hypotheses, rerun a failing test, etc.
  - Beware shell chains like `make foo && ./foo && echo PASS || echo FAIL`: if `make` itself fails (missing dep), the `||` branch prints "FAIL" and you misread a *build* failure as a *test* failure. Check the make exit status / log separately.
  - A `%check` assertion that fails under the distro flags but passes with upstream's hardcoded flags usually means honouring `%{optflags}` (`-O2`/LTO) exposed latent upstream UB (e.g. unaligned `u32` reads over a byte buffer, strict-aliasing violations) — bisect with `-fno-strict-aliasing` / `-fno-lto` / `-O0` in the chroot to pin it. A failure that differs *by arch* is instead a portability bug (e.g. default `char` signedness: signed on x86_64, unsigned on aarch64).
- **When dropping a patch, remove the file from package tracking, not just its spec reference.** In a classic osc checkout run `osc rm <patch>` (in a Git checkout, `git rm <patch>`) so the file is actually deleted from the package — then commit. A patch file left in the package dir but unreferenced by any `Source:`/`Patch:` tag is an *orphaned source*: `osc service runall source_validator` may pass locally, but **`factory-auto` will decline the SR** with `Attention, "<file>" is not mentioned in spec files as source or patch.` The same applies to any disabled patch — don't leave the file behind with a commented-out `PatchN:` unless you keep a real (commented) reference and intend to re-enable it (`PATCH-NEEDS-REBASE`). Beware malformed "disable by typo" comments like `#atch1:` / `#atch0:` (a `Patch` line with the leading `P` deleted) — these neither apply the patch nor count as a reference, so they orphan the file silently.
- **Proactively cross-check package files against spec references during cleanup — don't rely on `source_validator` to catch orphans.** Local `osc service runall source_validator` did **not** flag a tracked-but-unreferenced `*.patch` in practice, yet `factory-auto` declined on it server-side — so the validator passing is *not* proof there are no orphans. On any cleanup, diff the patch files actually present in the package dir against what the spec references, e.g.:
  ```
  comm -23 <(ls *.patch 2>/dev/null | sort) \
           <(grep -hoE '\S+\.patch' *.spec | xargs -n1 basename | sort -u)
  ```
  Anything printed is an orphan: either wire it back in (restore the `PatchN:` + a `# PATCH-…` tag and let `%autosetup -p1` apply it) or `osc rm` it. **Confirm with the user whether an orphaned patch should be applied or dropped** — a file someone deliberately committed may be a real fix that lost its `PatchN:` line (restore it) or genuinely dead (drop it), and the right call isn't always obvious from the file alone. Real case: the zathura stack had both kinds — mupdf's two `#atch`-typo'd patches were dead (dropped), while zathura's `no-parallel-xvfb.patch` was a lost reference; the user confirmed it no longer applied, so it too was `osc rm`'d. **A strong signal that an orphan is obsolete (already adopted upstream): it no longer applies.** Restore its `PatchN:` and build — if `%prep` fails with `N out of N hunk FAILED … .rej`, the source very likely already contains the change; confirm by grepping the extracted source for it, then **drop the patch** rather than force-fitting it. Real case: boringssl's orphaned `0001-Compile-for-RISC-V.patch` failed to apply because the pinned snapshot's `include/openssl/base.h` already carried the RISC-V `OPENSSL_*_BIT` defines — so it was dropped as obsolete, not restored.

## Changelog (`*.changes`)

https://en.opensuse.org/openSUSE:Creating_a_changes_file_(RPM)

- openSUSE keeps the changelog in a separate `<name>.changes` file, not inside `%changelog` in the spec.
- Use `osc vc` to add a new entry — it stamps the date/email correctly.
- Newest entry at the top. Entries are separated by a line of 67 dashes. Wrap text at 67 columns.
- **First-level bullet `-`, second-level `*`. No third level.** The `/usr/lib/build/changelog2spec` helper specifically recognizes these two symbols for auto-reindenting, and yast2-qt renders deeper levels badly — so never use a third level. **Every item is a bullet** — never write a `.changes` entry body as a bare prose paragraph; a single note still gets a leading `- `, and continuation lines wrap indented under their bullet.
- Never edit an existing entry once the package has been checked in to an official project (only trivial typo / bug-reference fixes are acceptable).
- Reference bugs as `bsc#NNNNNN` (SUSE bugzilla) or `boo#NNNNNN` (openSUSE). Avoid `bnc#`.
- **HARD RULE — CVEs need full identifiers.** Any mention of fixed security issues MUST cite the literal `CVE-XXXX-NNNN` id(s), one per fix (ideally paired with a `bsc#`/`boo#`). A vague "fixes multiple CVEs" / "security hardening (CVEs)" bullet without the actual IDs is unacceptable — the SUSE/openSUSE security tracker keys off the literal CVE strings.
- **HARD RULE — proactively search upstream for CVE IDs on every version bump.** Don't wait for release notes to spell them out; mine the commit range yourself. Clone/fetch upstream and grep between the old and new tag:
  ```
  git clone --filter=blob:none <upstream-url> /tmp/x && cd /tmp/x
  git log <oldtag>..<newtag> --pretty='%s%n%b' | grep -ioE 'CVE-[0-9]{4}-[0-9]{4,7}' | tr a-z A-Z | sort -uV
  ```
  Also scan the GitHub release page and any `SECURITY`/`NEWS`/`ChangeLog` file. For each ID: (1) confirm it's a *fix* by mapping it to its commit subject (`git log <range> -i --grep=<cve> --pretty=%s`), and (2) confirm it isn't already credited in an earlier `.changes` entry (`grep -ioE 'CVE-[0-9]{4}-[0-9]{4,7}' <pkg>.changes`). List every genuinely-new ID in the update entry, wrapped at 67 cols. Only when a thorough search yields nothing concrete do you fall back to a factual, non-CVE description (e.g. "many crash and memory bugs fixed via OSS-Fuzz") — never claim CVEs you can't name.
- **Don't credit bundled-component bumps the package doesn't use.** If the spec builds against the system copy of a library (e.g. `-DOPENBABEL_USE_SYSTEM_INCHI=ON` with `BuildRequires: inchi-devel`), omit upstream's "bundled X updated to N" note from the `.changes` — it's misleading because that bundled copy isn't shipped.
- Summarise what changed for an openSUSE *user* — don't dump upstream's full release notes, and don't reference the upstream changelog by URL alone. Keep it to ~10–15 lines on big releases.
- **When upstream ships no `NEWS`/`ChangeLog`/release-notes (common for libraries that just track another project's release train, e.g. swift-corelibs-libdispatch), mine the commit range and filter to packaging-relevant subjects.** Pull the range with the compare API (`curl -sS "https://api.github.com/repos/<o>/<r>/compare/<oldtag>...<newtag>" | jq -r '.commits[].commit.message'` or `gh api ... --jq ...`), then grep the subject lines for the categories a packager/user cares about — new platform/libc support (`Musl`, `FreeBSD`, `Android`), arch (`riscv`, `ARM64`), build-system (`CMake`, `install dirs`, `build id`), and notable fixes — and turn those into a handful of concrete bullets. This yields a verbose-yet-honest entry without inventing anything and surfaces the same spec-affecting items the up-front change-extraction rule wants. (Real case: libdispatch 5.6.3→6.1.1 had no NEWS file; the 90-commit compare yielded Musl support, Linux build IDs, un-hardcoded CMake install dirs, Android `posix_memalign` fallback, and Windows fixes.)

## Systemd

https://en.opensuse.org/openSUSE:Systemd_packaging_guidelines

- Install unit files under `%{_unitdir}` (system) or `%{_userunitdir}` (user) — never `/etc/systemd/system`.
- Do not mark unit files as `%config`.
- Do not require systemd by hand; use the `%service_*` macro family from `systemd-rpm-macros` (already pulled in by rpm-build) in `%pre`/`%post`/`%preun`/`%postun`. Call each macro at most once with all unit names as arguments.
- For sockets, use `%service_del_postun_without_restart` instead of `%service_del_postun`.
- To enable a service by default, submit a change to `systemd-presets-branding-openSUSE` (vendor opt-in) or `systemd-presets-common-SUSE` (truly required) — do not enable it from the spec itself.
- For pkgconfig deps: `BuildRequires: pkgconfig(libsystemd)` for the library, `pkgconfig(systemd)` for the tools.

## Sysusers / system users

- Use `sysuser-tools` + `%sysusers_requires` + `%sysusers_generate_pre` (see the wiki for the boilerplate). Users created by packages are never removed.
- **When upstream's systemd unit adopts `DynamicUser=yes` (+ `AmbientCapabilities=` for low-port binding), drop the whole static-user apparatus on the next bump.** A daemon's `.service.in` switching to `DynamicUser=yes` means systemd allocates a transient user at runtime, so the package no longer needs a static system user: drop the `sysuser-tools` BuildRequires, the `%sysusers_requires`/`%sysusers_generate_pre` calls, the shipped `*-user.conf` sysusers `Source` (and its `%{_sysusersdir}` install + `%files` line), and any downstream "disable upstream's useradd/groupadd" patch (upstream's Makefile usually drops the `useradd`/`groupadd` in the same release). Likewise `AmbientCapabilities=CAP_NET_BIND_SERVICE` in the unit replaces a `setcap` on the binary — drop any setcap handling. (Real case: nqptp 1.2.8 moved its service to `DynamicUser=yes` + `AmbientCapabilities`, letting the spec shed the sysusers file, `sysuser-tools`, the `%sysusers_*` macros, and the `disable-user-group-generation.patch`.) Caveat: if the daemon shares IPC (e.g. `/dev/shm`) with another service that runs as a *different* user, confirm the shared object's permissions still allow cross-user access under DynamicUser — but trust it when the *same upstream* maintains both ends (they test that combination).
- **A version bump can change *where* the systemd unit installs — re-check before keeping a manual relocation.** Older autotools/cmake often dropped the unit in `%{_libdir}/systemd/system` (→ the spec `mv`'d it to `%{_unitdir}`); a newer release may switch to the proper `systemdsystemunitdir` (`/usr/lib/systemd/system` = `%{_unitdir}`) and install there directly — at which point the spec's `mv %{buildroot}%{_libdir}/systemd/system/...` fails with `mv: cannot stat … (No such file or directory)`. Drop the now-obsolete relocation. (Real case: nqptp 1.2.8 — configure reported `systemd unit directory: ${prefix}/lib/systemd/system`, so the lib64 `mv` had to go.)
- Default shell for system users is `-` (becomes `/usr/sbin/nologin`); never `/bin/false` or `/bin/true`.
- **When upstream starts shipping its own sysusers.d file (or `.desktop`, systemd unit, tmpfiles.d, udev rule, completion, …), prefer the upstream-shipped file over a downstream-maintained `Source`** — drop the distro copy unless there's a really good, documented reason to keep it (e.g. the upstream file is broken or omits something the distro needs). Carrying a downstream duplicate of a file upstream now installs is redundant and tends to drift or, for same-named files, conflict. This commonly surfaces on a version bump: a project that previously had no sysusers file adds one, and the spec's hand-shipped `system-user-<name>.conf` becomes dead weight. Mechanics: point `%sysusers_generate_pre` (and `%files`) at the path upstream's `make install` produces (e.g. `%{_sysusersdir}/<name>.conf`), drop the manual `install` of the old `Source`, and `osc rm`/`git rm` the now-orphaned downstream file. (Real case: libstoragemgmt 1.10.3 began shipping `%{_sysusersdir}/libstoragemgmt.conf` via `packaging/daemon/user_group/`; the spec switched to it and dropped its own `system-user-libstoragemgmt.conf`. Bonus: upstream's used the `u!` *locked-account* form, which is the recommended style anyway.) This is the same "prefer upstream" judgement as the `.desktop` cases under "Desktop files / GUI apps".

## Desktop files / GUI apps

- GUI apps need a `.desktop` file following https://standards.freedesktop.org/desktop-entry-spec/. `Icon=` must be a basename (no extension).
- **A package that installs its icon to `%{_datadir}/pixmaps` (not into a `…/icons/hicolor/…` theme dir) does *not* need `Requires: hicolor-icon-theme` (nor the BuildRequires)** — pixmaps is a flat fallback dir owned by `filesystem`, unrelated to the hicolor theme. An unversioned `Requires: hicolor-icon-theme` there trips rpmlint's `E: branding-requires-unversioned hicolor-icon-theme`; dropping the dependency is both correct and clears the error. Only depend on `hicolor-icon-theme` when you actually install into its theme directories. (Real case: lshw-gui shipped its icon in `%{_datadir}/pixmaps` yet required `hicolor-icon-theme` — dropped.)
- **HARD RULE: `%suse_update_desktop_file` is deprecated and must always be removed** (rpmlint flags `suse-update-desktop-file-deprecated`). Never keep it, in either form:
  - The **`-i` (in-place) form** edits an existing upstream `.desktop` — replace it by declaring that file as a `Source` and installing it directly (`install -Dm 0644 %{SOURCE1} %{buildroot}%{_datadir}/applications/<name>.desktop`).
  - The **`-c` (create) form** generates a `.desktop` from macro arguments — replace it by **shipping a hand-written `.desktop` as a `Source`** with the same fields (`[Desktop Entry]` / `Type` / `Name` / `GenericName` / `Exec` / `Icon` / `Categories=…;`) and installing it the same way. To recover the exact fields the macro produced, read the generated file from a prior build's `SOURCES/<name>.desktop` (or the BUILDROOT copy).
  - In both cases **drop `BuildRequires: update-desktop-files`** (it only provided that build-time macro — see the `Requires(post)` rule above), and don't add runtime desktop-database scriptlets (file triggers handle the cache).
  - If the spec is an altcoin-style template that seds `%{base}`→`%{name}` across its sources, write the shipped `.desktop` with the *base* tokens (e.g. `bitcoin`/`Bitcoin`) so the existing sed templating keeps working, and slot it into the reserved `Source` number the sed block already references. (Real case: bitcoin's `%suse_update_desktop_file -c` replaced by a shipped `bitcoin-qt.desktop` as `Source2`, which the spec's `%if !is_base` sed block already expected.)
- **HARD RULE: `update-desktop-files` is a *build-time* macro provider — never put it in `Requires(post)`/`Requires(postun)`.** It ships the build-time `%suse_update_desktop_file` macro; it is not a runtime dependency, so `Requires(post): update-desktop-files` / `Requires(postun): update-desktop-files` are simply wrong. The runtime cache-update scriptlet macros come from *different* packages: `%desktop_database_post`/`%desktop_database_postun` (run `update-desktop-database`) are from **`desktop-file-utils`**; `%icon_theme_cache_post`/`%icon_theme_cache_postun` are from **`hicolor-icon-theme`**. So if a scriptlet is genuinely needed, the `Requires(post)`/`Requires(postun)` must name *that* provider, never `update-desktop-files`.
- **Prefer dropping the desktop/mime/icon-cache `%post`/`%postun` scriptlets entirely — modern openSUSE handles them via file triggers.** `desktop-file-utils`, `update-desktop-files`, and `hicolor-icon-theme` all ship **file triggers** that automatically rebuild the desktop database, MIME cache, and icon cache whenever files land in `/usr/share/applications`, `/usr/share/mime`, or an icon theme dir. So the old manual `%post qtX`/`%postun qtX` calling `%desktop_database_post`/`%icon_theme_cache_post` (plus their `Requires(post)`/`Requires(postun)`) are redundant and should be removed. (Doubly so when the `.desktop` has no `MimeType=` — `update-desktop-database` then does nothing useful anyway.) Real case: bitcoin's `qt` subpackage carried `Requires(post)/Requires(postun): update-desktop-files` + `%post/%postun` running `%desktop_database_post` — all dropped in favour of the file triggers.
- **`%suse_update_desktop_file -i <name>` hides an orphaned source.** The `-i` form reads `<name>.desktop` from the rpm **SOURCES dir** — where osc copies *every* file in the package dir — so the build succeeds even when the `.desktop` is **not** declared as a `Source:`. But that undeclared file is an *orphaned source*: `factory-auto` declines with `Attention, "<name>.desktop" is not mentioned in spec files as source or patch`. Fix by declaring it (`Source1:  <name>.desktop`); migrating to the explicit-install form above does that anyway. (Real case: `vorta` — `vorta.desktop` was used only via `%suse_update_desktop_file -i vorta`, undeclared, and the SR was declined for the orphan; declaring it as `Source1` + explicit install fixed both the orphan and the deprecation.) This is the same orphaned-source trap as the patch case in the Patches section — any package file the build consumes must be a declared `Source`/`Patch`.

## Web apps

- Install under `/etc`, `/usr`, `/var` like any other package. Do *not* put files in `/srv` (admin-owned per FHS). Creating an empty `/srv` dir is acceptable if the default config points there.

## Language-specific quick reference

When packaging language-specific software, fetch the matching wiki page for the full rules — these summaries cover only the structural choices.

- **Vendored-dependency packages (Go / Rust / Node / any language whose deps are fetched at build time): generate the vendor bundle with a `_service`, the thin-provisioning-tools way — do NOT hand-vendor, and do NOT download a language toolchain yourself to run `go mod vendor` / `cargo vendor` by hand.** The canonical setup is a `_service` file (services `mode="manual"`) that fetches the source and produces the vendor tarball, whose output you then commit as *tracked sources* and consume from the spec. Worked model (thin-provisioning-tools, a Rust pkg — the same shape applies to Go with `cargo_vendor`→`go_modules`):
  ```xml
  <services>
    <service name="obs_scm" mode="manual">      <!-- fetch upstream at the tag -->
      <param name="url">https://github.com/…/foo.git</param>
      <param name="scm">git</param>
      <param name="revision">refs/tags/v1.3.2</param>
      <param name="versionformat">@PARENT_TAG@</param>
      <param name="versionrewrite-pattern">v(.*)</param>
      <param name="changesgenerate">enable</param>
    </service>
    <service name="tar" mode="manual"/>
    <service name="recompress" mode="manual"><param name="file">*.tar</param><param name="compression">zst</param></service>
    <service name="set_version" mode="manual"/>
    <service name="cargo_vendor" mode="manual">   <!-- Go: name="go_modules" -->
      <param name="srcdir">foo</param>           <!-- Go: point at the dir holding go.mod -->
      <param name="compression">zst</param>
      <param name="update">true</param>
    </service>
  </services>
  ```
  Then in the spec: `Source0: %{name}-%{version}.tar.zst`, `Source1: vendor.tar.zst`, and `%autosetup -a1` (or `-p1 -a1`) to unpack the vendored tree next to the source. Commit the generated `%{name}-%{version}.tar.zst` **and** `vendor.tar.zst` (plus `_service` / `_servicedata` / `*.obsinfo`) as real package files. Run the services with `osc service manualrun` (the host needs the matching `obs-service-cargo` / `obs-service-go_modules` and toolchain) **or** let them run server-side — either way the point is that the *service* produces `vendor.tar.*`, never a by-hand vendor dir. This is the preferred way to create any new vendored-deps package; the per-language rows below give the build macros.
- **Require a compiler/runtime by *versioned virtual provide*, not by pinning the implementation package.** `BuildRequires: go >= 1.25` (the `go1.N` packages each `Provides: go = 1.N`, so this floats forward to 1.26+) — never `BuildRequires: go1.25`, which nails it to one compiler package. Same idea for other versioned virtuals (`java >= 1.x`, etc.).
- **Packaging a Falco / libsinsp *plugin* (a `lib<name>.so` loaded by libsinsp): the plugin links the header-only plugin SDK only — NOT libsinsp.** A plugin is `dlopen`'d by the inspector at runtime, so its build needs `plugin-sdk-cpp` (+ whatever else its own code links), and it picks up the *system* libsinsp at load time. The trap is that the upstream build often pulls a **from-source libsinsp** in only for its **unit tests** (a `sinsp_test_support` target) — so building with the tests **off** (`-DENABLE_TESTS=OFF` or equivalent) drops the entire nested-libsinsp build and lets the plugin build against the shared stack. Install the `.so` under `%{_libdir}/falcosecurity/plugins/`. (Real case: falcosecurity-container-plugin — `include(libs)` lived only in `test/CMakeLists.txt`; tests off → no libsinsp needed, links just plugin-sdk-cpp + fmt + shared re-flex + the Go worker.)

### Packaging a multi-package stack (a new dep cone) — build in a home: project, in parallel

When a new package drags in a chain of unpackaged deps (e.g. a Python app needing 10+ new modules), **do not build them one-at-a-time with local `osc build`** — local builds are serial (one VM at a time) and you'll wait hours. Instead create a scratch project (`osc meta prj -e home:<user>:<stack>`, add an `openSUSE_Factory` repo with path `openSUSE:Factory/standard`, arch `x86_64`) and **commit each prepped package into it**; OBS schedules them all in parallel and re-triggers dependents automatically as each leaf publishes. Workflow:
- **Determine build order leaves-first** but you don't have to wait between commits — commit everything you can, and the ones whose deps aren't published yet sit at `U` (unresolvable) and auto-rebuild the moment the dep appears. A transient `U` right after committing a dep is normal (publish lag), not an error — only worry if it persists after the dep shows `.` succeeded.
- **Monitor with `osc results <prj>`** (one column per package, dependency-sorted) — `.`=succeeded, `F`=failed (build ran, spec/code error → read the log), `U`=unresolvable (a `BuildRequires` isn't available/published yet), `B`=broken (source-level problem, often **"no source uploaded"** = you `osc add`'d files but never `osc ci`'d them — just commit), `b`=blocked (waiting on the scheduler), `%`=building.
- **Read a remote build log with `osc rbl <prj> <pkg> openSUSE_Factory x86_64`** (remote-buildlog; `osc buildlog` is the same for the current dir). Grep it for `error|fail|nothing provides|unpackaged|Unknown build hook`. A `B`-broken package has **no** log (`HTTP 404 … has no logfile`) — check `osc api /build/<prj>/<repo>/<arch>/<pkg>/_status` and `osc ls <prj> <pkg>` instead.
- Keep `osc st` for *local* working-copy state (uncommitted `A`/`M` files). Be aggressive about firing off independent `osc rbl` / `osc results` / fetches in parallel rather than serially.
- Only fall back to a local `osc build` when you need to iterate fast on one stubborn package in isolation; otherwise let the project build.
- **Read the `osc rbl` log of the *successful* packages too, not only the failed ones** — in a server-built cone a `.` (succeeded) only means rpmlint badness stayed under the 999 threshold and the build completed; it does not mean the log is clean. Grep each green package's log for the `RPMLINT report` section and any `W:`/`E:` lines, for sub-threshold rpmlint findings worth fixing (e.g. `devel-file-in-non-devel-package`, `non-executable-script`, weak-dependency suggestions), for compiler warnings that signal real bugs (`-Wmaybe-uninitialized`, `-Wimplicit-function-declaration`), and for **missed/incorrectly-auto-satisfied dependencies** (a runtime import that resolved only because a build dep happened to pull it in transitively). This is the server-side analogue of the "always read rpmlint after every build" hard rule — when you never ran the build locally, the remote log is the only place those problems show, so review it for every package in the cone before submitting. Filter the environmental noise this build farm always emits (the `polkit-default-privs … scriptlet failed`, `Disabling truststore because of missing certificates`, and `cycle: …` lines are not your package's problem).
- When the whole cone is green, SR each package to `devel:languages:python` (leaves first so deps exist in the target). If the user wants to maintain them, add the user to each package `_meta` (`<person userid="…" role="maintainer"/>`) **and** say so in the SR message.

### Python singlespec deep-dive (what `py2pack` won't get right)

`py2pack fetch <name> && py2pack generate <name>` scaffolds a spec but the result always needs hand-cleanup. The recurring gotchas, all learned building the FastMCP cone:
- **Pick the build backend's helper modules from `pyproject.toml`, not guesswork.** Read `[build-system] requires` *and* the `[tool.hatch.*]` hooks: `[tool.hatch.version] source = "vcs"` → add `BuildRequires: %{python_module hatch-vcs}` (the Factory package is `python-hatch_vcs` with an underscore, but `%{python_module hatch-vcs}` resolves via Provides); `[tool.hatch.metadata.hooks.fancy-pypi-readme]` → `%{python_module hatch-fancy-pypi-readme}`; `uv_build` backend → `%{python_module uv-build}`; `uv-dynamic-versioning` → `%{python_module uv-dynamic-versioning}`; poetry-core, setuptools-scm, etc. likewise. A missing build hook fails late as `hatchling.plugin.exceptions.UnknownPluginError: Unknown build hook: vcs` / `metadata-generation-failed`.
- **Derive runtime deps from the sdist's `PKG-INFO` `Requires-Dist`, not the README.** Base (unconditional) deps + whichever `extra ==` groups you actually need (e.g. a meta-package `foo[client,server]` → union the `client` and `server` extras). Mirror each as **both** `%{python_module X}` in `BuildRequires` **and** `python-X` in `Requires` (runtime deps must also be build deps or `%check`'s import fails with `ModuleNotFoundError`). `pydantic[email]` → also pull `email-validator`.
- **Casing false-positives in Factory dep resolution** — try the canonical PyPI casing: `Authlib` (not authlib), `PyYAML` (not pyyaml), `typing_extensions` (underscore), `Pygments`. If `%{python_module x}` reports unresolvable but the package clearly exists, it's almost always casing.
- **`%check`: use a manual import smoke test, not `%pyproject_check_import`.** `%pyproject_check_import` can expand to a shell line that dies with `/var/tmp/rpm-tmp: line N: fg: no job control`, and full test suites drag in unpackaged pytest plugins (`pytest-asyncio`, `aiomisc-pytest`, `--cov`/addopts). Prefer `%python_expand PYTHONPATH=%{buildroot}%{$python_sitelib} $python -B -c "import MODULE"` (use `%{$python_sitearch}` for C extensions). Note the module's *import* name can differ from the dist name (src-layout: `uncalled_for`, namespace: `key_value.aio`).
- **The `python-bytecode-inconsistent-mtime` rpmlint error is systemic in this build env** (appears even without vendoring). Fix in `%install` on every Python package: `%python_expand $python -m compileall -q -f -o 0 -o 1 --invalidation-mode unchecked-hash %{buildroot}%{$python_sitelib}/MODULE`. The `-o 0 -o 1` is required — without both, only the `.pyc` (not `.opt-1.pyc`) gets the hash-based invalidation and half the errors remain. Use `$python` (shell var), never `%{$python}` (expands literally to `%{python311}` → `fg: no job control`).
- **Console scripts (a `[project.scripts]` entry → `/usr/bin/NAME`) need the singlespec alternatives dance**, else the build fails with `Installed (but unpackaged) file(s) found: /usr/bin/NAME`: in `%install` add `%python_clone -a %{buildroot}%{_bindir}/NAME`; add `%post`→`%python_install_alternative NAME`, `%postun`→`%python_uninstall_alternative NAME`; and `%python_alternative %{_bindir}/NAME` in `%files`.
- **Verify the `%doc` filename against the tarball** (`tar tzf … | grep -i readme`) — `README.md` vs `README.rst` mismatch fails at packaging with `File not found: …/README.rst`. Same for `%license`: omit it entirely if the sdist ships no LICENSE/COPYING file (many `*-slim` redistribution packages don't).
- **C-extension modules** (e.g. caio): `BuildRequires: %{python_module devel}` + `gcc`, drop the bundled `*.c`/`*.h` the wheel ships alongside the `.so` (`find %{buildroot} -name '*.c' -delete; … '*.h' …`, else `devel-file-in-non-devel-package` badness), and use `%{python_sitearch}` (arch-specific) throughout instead of `%{python_sitelib}`.
- **Python-version floor → flavour selection.** `requires-python >= 3.13` → `%define pythons python313` (single flavour); a `< 3.11` conditional dep (`typing-extensions`/`tomli` guarded by `python_version<'3.11'`) maps to `%if %{python_version_nodots} < 311` — but the built flavours are all ≥311 so those branches never fire; keep them for correctness only.
