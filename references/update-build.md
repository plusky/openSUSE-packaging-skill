# Block 2 — Update, build, clean up

## Pre-flight: is this update already done or in flight? (HARD RULE — do this BEFORE any edit/branch/build)

**Before you `osc branch`, bump a `Version:`, regenerate a vendor tarball, or start a (possibly very expensive) build, confirm you are not redoing work the maintainer already did.** Repology/triage flags "outdated" against *published Tumbleweed*, which lags the devel project — the devel project (and an in-flight SR) may already carry the exact version you're about to package. Run **both** checks against the package's devel project (`osc develproject openSUSE:Factory <pkg>` to find it):

1. **Does the devel project already carry the target version?**
   ```
   osc api "/source/<devel>/<pkg>/<pkg>.spec" | grep -m1 '^Version:'
   osc api "/source/<devel>/<pkg>/<pkg>.changes" | sed -n '1,4p'   # top entry: version + author
   ```
2. **Is there already a submit request in flight for it — in *either* direction?** Check both incoming and outgoing:
   ```
   osc request list <devel> <pkg>        # lists SRs involving the pkg in that project
   # outgoing devel -> Factory (this is the one that lands the update):
   osc api "/request?view=collection&types=submit&states=new,review&project=openSUSE:Factory&package=<pkg>"
   ```
   - **Incoming** = someone's `home:…:branches/<pkg> -> <devel>` SR (a maintainer staging the bump into devel — e.g. a `home:favogt:branches…/python-uv -> devel:languages:python` "cleanup" SR). Even if the devel spec doesn't *yet* show the new version, an accepted/open incoming SR means it's already handled.
   - **Outgoing** = the `<devel> -> openSUSE:Factory` SR that actually delivers the update to Factory.

**Decision matrix:**

| devel has target version? | Factory SR (devel→Factory) exists (new/review)? | Action |
|---|---|---|
| yes | yes | **STOP — nothing to do.** The update is already on its way. Report the SR id; do not repackage, do not file a competing SR. |
| yes | **no** (stranded devel update) | **Don't repackage — just forward it:** confirm green (`osc results <devel> <pkg>`), then `osc sr <devel> <pkg> openSUSE:Factory`. One command, zero packaging. (See triage.md "stranded devel update", real cases openmopac/pspg.) |
| no | — | Proceed with the normal Block 2 update below. |

If the devel version already matches and you only have *cosmetic* spec-cleaner changes left, do **not** file a cleanup-only SR over an in-flight version SR — it races the version SR and earns a "conflict in file" decline. Hold the cleanup until the version SR lands, or drop it.

**Real case (the rule's origin):** asked to "update python-ruff (0.15.20) and python-uv (0.11.25)", an agent branched both, re-bumped, re-vendored, and ran two heavy native Rust builds — only to discover *afterwards* that both versions were **already committed to devel by the maintainers** (ruff by mimi_vx, uv by mimi_vx + fvogt) **and already submitted to Factory** (SR 1362328 ruff, SR 1362488 uv, both in review). The entire update+build was wasted; the only net delta was a cosmetic cleanup that would have conflicted with the in-flight SRs. A 30-second pre-flight `osc api .../<pkg>.spec | grep Version` + SR check would have caught it before any branch or build.

`scripts/preflight.sh` mechanizes this whole check (exit 0 PROCEED / 3 STOP / 4 FORWARD).

## Local builds (`osc build`)

- **HARD RULE: investigate builds locally by default; reach for the *remote* build (`osc rbl`/`osc results`/`br.opensuse.org`) only in the specific cases where local is impossible or irrelevant.** When a build fails or you need to read a build log, the first move is the *local* artefacts — the preserved build root at `/var/tmp/build-root/<repo>-<arch>/` and its saved log `/var/tmp/build-root/<repo>-<arch>/.build.log` (read it directly — **no sudo**; `scripts/build-summary.sh [repo-arch]` extracts the result, `%check`/ctest pass count, rpmlint badness + every E:/W: line, and the produced RPMs in one go. If a restricted profile ever blocks reading under `/var/tmp/build-root`, capture the build yourself — `osc build … 2>&1 | tee /tmp/osc-build.log` — and pass that file to the script; osc streams the identical log to stdout. See the rpmlint and `osc chroot` rules below). You already have the full log and a debugger-friendly root on disk from the build you just ran — querying the server for the same information is slower, depends on the server having scheduled/finished your commit, and reflects a *different* environment than the one you reproduced in. So diagnose, grep for errors, bisect compiler-flag hypotheses, and rerun failing tests **in the local root**, not over the wire. The remote build is the right tool only when: (a) the failure is *arch-specific* and you can't build that arch locally (e.g. a foreign arch needing an x86_64 host, or an arch your machine lacks — see the emulation notes below); (b) the user explicitly asks you to check/monitor the server-side build, or there is a concrete reason to (e.g. diagnosing a failure that reproduces *only* on OBS, not locally); or (c) you genuinely have no local checkout/build to inspect (triaging someone else's pasted failure URL). Outside those, prefer local every time — don't poll `osc rbl` for something sitting in `.build.log` on your own disk.
- **Monitoring server-side builds is NOT a default step — never poll `osc results`/`osc rbl` on your own initiative after a commit or SR.** Once the local `osc build` + `source_validator` are green you commit and (for Factory) file the SR immediately; the server rebuild is the bots'/maintainers' concern, not something to babysit. Watch the remote matrix only when the user asks, or when you have a specific reason to (a suspected OBS-only failure, an arch you couldn't build locally). This matches the "file the SR immediately, don't poll `osc results`" rule under Submit requests. Reporting "committed/submitted" is a complete outcome — don't append speculative remote-status checks.

- **Before working on an *already-checked-out* package, run `osc up` first to sync it with OBS.** A pre-existing `.osc/` working copy (one you did not just `osc co` this turn) can be behind the server — commits from collaborators, bots, or a previous session — so editing/building/diffing against a stale base risks a clash or silently reverting server-side changes at commit. Do it automatically on reuse; a fresh `osc co` is already current so it doesn't need it. (For a whole tree, `osc up` at each project-level checkout updates all the packages within it.)
- Default target architecture is the host's native arch (`uname -m`). Do not pick a non-native arch unless the user explicitly requests it — emulation through qemu-user is dramatically slower (observed: 19s native aarch64 vs ~240s emulated x86_64 for the same trivial package).
- If the chosen repo does not publish the native arch (e.g. `openSUSE_Tumbleweed` has no aarch64 in `osc repos`), prefer another repo that does (`openSUSE_Factory`, `16.0`, `15.6`, …) rather than building emulated.
- **Match the build project to the package's arch — `openSUSE:Factory:ARM/standard` publishes only `aarch64`/`armv7l`/`armv6l`, while `openSUSE:Factory/standard` publishes only `x86_64`/`i586`.** So on an aarch64 host: aarch64-capable packages build against `Factory:ARM` natively; an **x86-only package** (`ExclusiveArch: %{ix86} x86_64`, e.g. `iucode-tool` and other Intel-microcode tools) has **no aarch64 to build**, so build it against **`openSUSE:Factory standard x86_64`** — which runs *emulated* on the aarch64 host (slower, but it's the package's only arch). Targeting `Factory:ARM … x86_64` errors with `x86_64 is not a valid arch for the repository standard`.
- **`osc build` arg order: `osc build [--alternative-project <prj>] <repository> <arch> <build_descr>`.** The project is **not** a positional — pass it via `--alternative-project` (or rely on the checkout's `.osc` project). Writing `osc build openSUSE:Factory standard x86_64 foo.spec` (project as a positional) fails with `Too many arguments`; the correct form is `osc build --alternative-project openSUSE:Factory standard x86_64 foo.spec`.
- **The repo *name* set differs per devel project — never assume a fixed one; run `osc repos <prj>` first.** Different devel projects expose different repository names: e.g. `science` and `server:mail` build against **`openSUSE_Tumbleweed`** (which has aarch64), while `Archiving` builds against **`openSUSE_Factory`** (also aarch64), and a git/`Factory` checkout uses **`standard`** under `openSUSE:Factory[:ARM]`. Passing a name the project doesn't have errors with `<name> is not a valid repository, use one of: …` (and a wrong arch with `<arch> is not a valid arch for the repository <name>`). So before the first build in an unfamiliar project, run `osc repos <prj>` and pick a repo+arch row that lists your native arch.
- **PREFER building against Factory via `--alternative-project`, rather than the devel project's own repo — stick to Factory.** A devel project may only expose Leap/Tumbleweed/`16.0` repos (e.g. `server:irc` has `openSUSE_Tumbleweed`/`16.0`/`15.6` but **no** Factory repo), and building against those tests a *different* target than where the SR goes (`openSUSE:Factory`). For a representative, consistent local build, pass **`osc build --clean --alternative-project=openSUSE:Factory standard <arch>`** (use **`openSUSE:Factory:ARM`** for native aarch64, since plain `openSUSE:Factory/standard` publishes only `x86_64`/`i586`). This builds against Factory's config regardless of what repos the devel project happens to list, so you catch what Factory will catch. Fall back to a devel-project repo (per the `osc repos` bullet above) only when Factory genuinely can't build it (e.g. the package isn't in Factory, or needs a Leap-only dependency). (Real case: znc in `server:irc` — built against `--alternative-project=openSUSE:Factory:ARM standard aarch64` instead of the project's `16.0` Leap repo.)
- **Multibuild packages (`_multibuild` file) — build with `-M <flavor>`, and verify *every* flavor.** A package with a `_multibuild` listing `<package>` flavors builds each flavor as a separate result; the spec keys off `@BUILD_FLAVOR@` (often `ExclusiveArch: do-not-build` when the flavor is empty, so a plain `osc build` does nothing useful). Pass the flavor as the **`-M`/`--multibuild-package` option, not a positional** — `osc build <repo> <arch> foo.spec db` fails with `Too many arguments`; the correct form is `osc build -M db <repo> <arch> foo.spec`. After a spec change that touches shared `BuildRequires` or `%build`/`%install` logic, build **all** flavors (loop over the `_multibuild` entries), not just one — a change can pass one backend and break another. (Real case: bogofilter — flavors `db`/`kyotocabinet`/`sqlite3`, all three rebuilt to confirm the cleanup.)
  - **Variant: a package with *multiple spec files* (a `<name>.spec` plus a companion `<flavor>.spec`, the flavor selected by `_multibuild`, often used to break a build cycle).** On a **version bump you must edit `Version:` in EVERY spec file**, not just the main one — the flavor specs share the same `Source` tarball. Miss one and `source_validator` (which validates *all* spec files in the package) aborts the build before it even starts with `ERROR: <name>-<OLDversion>.tar.gz mentioned in spec file is not tracked` coming from the un-bumped sibling. Each spec also has its **own `<flavor>.changes`** — add a matching changelog entry to each (respecting the never-amend-others rule per file). (Real case: libpsl 0.21.5→0.22.0 — `libpsl.spec` + `psl-make-dafsa.spec`; bumping only the first failed source_validator until `psl-make-dafsa.spec` was bumped too.)
- **HARD RULE: after every build, read the rpmlint output at the end of the build log — always, no exceptions.** That's where post-build package issues surface (file permissions, debuginfo problems, missing `%check`, RPATH errors, etc.), and crucially `osc build` writes the RPMs *before* running rpmlint, so "RPMs were produced" is **not** proof of a clean build. Always locate and read the final `N packages and ... checked; N errors, N warnings, ... N badness` line: any badness over the project threshold (999) means the **server** build will abort even though your local build wrote RPMs. Treat each warning as a thing to investigate, not noise to discard, and report the rpmlint summary back as part of "the build passed". See https://en.opensuse.org/openSUSE:Packaging_checks for explanations. For spec-file style use `spec-cleaner` (see above) — rpmlint and spec-cleaner check different things, so both must be run.
- For `osc build`, prefer `--no-verify --trust-all-projects` only if the user has already accepted that trade-off in this project; otherwise build with default verification.
- After a failed build, the build root is preserved at `/var/tmp/build-root/<repo>-<arch>/home/abuild/rpmbuild/BUILD/<pkg>-<ver>-build/<pkg>-<ver>/`. Use it to reproduce and diagnose runtime failures (e.g. running the failing binary directly, inspecting intermediate files) rather than re-running the full build to gather one more line of output.
- **HARD RULE: when *rerunning* `osc build` after a failed build, always pass `--clean`.** A failed build leaves the root in a partial/dirty state — half-installed deps, a stale CMake cache (old `-D` values, cached `find_package` results), leftover object files — and reusing it can mask the real fix or surface misleading errors that have nothing to do with your change. `osc build --clean ...` rebuilds the root from a well-defined state so the rerun actually tests what you changed. The *first* build of a package in a session can reuse an existing root (faster), but the moment a build fails, every subsequent rerun for that package uses `--clean` until it goes green. (If iterating only on `%check`/`%install` after a known-good `%build`, the preserved root is still useful for `osc chroot` reproduction — that's inspection, not a rerun.)
- **To enumerate installed files (for `%files` clean-up, replacing a `%{_bindir}/*` glob with explicit entries, etc.), query the previous build's RPM with `rpm -qlp`** — do not `find` or `ls` the `BUILDROOT/` tree. The buildroot is owned by the `abuild` user (uid 399) and often unreadable; the RPM under `RPMS/<arch>/` is authoritative for "what got installed" and has no permissions hassle. Example:
  ```
  rpm -qlp /var/tmp/build-root/<repo>-<arch>/home/abuild/rpmbuild/RPMS/<arch>/<name>-*.<arch>.rpm \
      | grep -E "^/usr/bin/|/man[15]/" | sort
  ```
  For a multi-subpackage build, run once per subpackage RPM in the same dir (each has its own `.rpm`).
- **Not every rpmlint `E:` is actionable — weigh it against what the package *does*.** A package that compiles software *at runtime* (e.g. kcbench, which builds a Linux kernel to benchmark the machine) legitimately `Requires:` `*-devel` packages (`libelf-devel`, `openssl-devel`), which trips `E: devel-dependency` and `E: explicit-lib-dependency`. Those are false positives here — the headers really are needed at run time — and rpmlint scores them at **badness 1 each**, far below the 999 abort threshold, so the build still passes. Don't "fix" a correct dependency to silence rpmlint; confirm the badness total stays under threshold and move on. (Contrast with the relative-RPATH error, badness 10000, which *must* be fixed.)
- Maintainer/owner enumeration queries (owner-search API, `osc whois` caveat, `%2B` encoding, person-index queries): see `references/triage.md`.
- **`python-bytecode-inconsistent-mtime` rpmlint errors are usually a *local-build-only* artifact, not a package defect.** A package that doesn't yet exist on the server has no release number locally, so `osc build` can't set `SOURCE_DATE_EPOCH` (the log warns `could not set SOURCE_DATE_EPOCH, ensure BUILD_RELEASE is set` and `SOURCE_DATE_EPOCH is not set: skipping … normalization`); without it the installed `.py` mtimes and the byte-compiled `.pyc` embedded timestamps drift by a second or two → 16 spurious `python-bytecode-inconsistent-mtime` errors. The server sets it from the changelog date, so they don't appear there. To reproduce server conditions locally, build with **`osc build --release 1 …`** (sets `BUILD_RELEASE` → `SOURCE_DATE_EPOCH`); the errors vanish, confirming they were the artifact. (Real case: a new `uv_build`-backend Python package on IBS.)
- **New `uv_build` Python backend** (`pyproject.toml` `build-backend = "uv_build"`): add `BuildRequires: %{python_module uv-build}` (the buildroot needs the backend for `%pyproject_wheel`'s `--no-build-isolation`). `python-uv-build` is in Factory but **not in Leap 16.0/SLE15** yet, so such a package is currently Tumbleweed-only there. Also: `%pyproject_check_import` may not exist in an older Factory snapshot's `python-rpm-macros` (it expands literally and fails `%check` with `fg: no job control`) — fall back to a manual import test `%python_expand PYTHONPATH=%{buildroot}%{$python_sitelib} $python -c "import <module>"` (note: the interpreter inside `%python_expand` is the shell var `$python`, **not** a `%{$python_exec}` macro). For a Python ≥ 3.13-only package, pin `%define pythons python313` so the singlespec doesn't try (and fail to resolve) the python311 flavor.
- **For a *native* (arch) Python package, use `%pytest_arch` in `%check`, not `%pytest`.** `%pytest` sets `PYTHONPATH` to the buildroot's `sitelib`, but a compiled C/Cython extension installs into `sitearch` — so a test that `import`s the module fails at collection with `ModuleNotFoundError: No module named '<mod>'` and aborts `%check`. `%pytest_arch` points `PYTHONPATH` at the installed `sitearch` where the `.so` lives. (Symmetry: `%pytest` for noarch/pure-python, `%pytest_arch` for packages that build a native extension and ship `BuildArch`-less / sitearch files.) Real case: `python-siphash24` — a meson-python Cython extension; `%pytest test.py` failed to import `siphash24` until switched to `%pytest_arch test.py` (then 13 tests passed). The same sitearch-vs-sitelib distinction applies to a manual import test and to the `%files` globs (`%{python_sitearch}/…` for the `.so` + dist-info).
- **A Python test that spawns a fresh interpreter (`subprocess.run([sys.executable, "-c", "import pkg…"])`) needs the *installed* package on `PYTHONPATH` — but adding the buildroot blindly makes pytest *also* collect `src/` and abort.** `%pytest`'s own path setup doesn't propagate to a child interpreter, so such tests fail (`returncode 1`, empty stdout). Fix with **both**: `export PYTHONPATH="%{buildroot}%{python3_sitearch}"` (so the subprocess imports the installed package) **and** point pytest at the test dir only — `%pytest … tests` — otherwise pytest collects `src/pkg/*.py` and dies with `import file mismatch … not the same as the test file we want to collect` (27 collection errors). (Real case: reuse 6.2.0 `TestEncodingModule`.)
- **Python docs built with `sphinxcontrib.apidoc`/`autodoc` import the package and `intersphinx` fetches inventories over the network — both break the offline OBS build.** autodoc imports each module, but `conf.py` rarely adds `src/` to `sys.path`, so `make -C docs man` dies with `ModuleNotFoundError: No module named '<pkg>'` → `export PYTHONPATH="$PWD/src"` before it (the package's `__init__` typically falls back to a hardcoded version when not installed, so no install needed). And `intersphinx_mapping = {"python": ("https://docs.python.org/3", None)}` can't fetch `objects.inv` offline; if the Makefile passes `--fail-on-warning`/`-W` (`SPHINXOPTS ?= --fail-on-warning`) that lone warning becomes fatal — override `%make_build -C docs man SPHINXOPTS=` to drop it. (Real case: reuse 6.2.0.)
- **Avoid full-project checkouts (`osc co <prj>` with no package) when you only need a few packages.** A bare project checkout registers/pulls *every* package in the project — that's tens of GB and hundreds of dirs for big projects (e.g. `hardware`, `science`), makes `osc up` crawl, and can **fill the disk** (ENOSPC mid-update). It also makes `osc up` try to check out *newly-added* packages, including **broken links**: `Link "<prj>/<pkg>" cannot be expanded: <devel>/<pkg>: package '<pkg>' does not exist` + `Use "osc repairlink" to fix merge conflicts` — a server-side bad/renamed link (not your local edit), which `osc repairlink`/`osc pull` can't fix because the target is gone. To work on just your packages, **check out each individually** (`osc co <prj> <pkg1>; osc co <prj> <pkg2>; …`) — that yields a *targeted* project working copy whose `osc up` only updates those packages and never pulls the broken link. To trim an existing full checkout down: confirm `osc status` shows no real local changes (only `!`/missing or `S`/service lines — no `M`/`A`/`D`/`C`), then `rm -rf <prj>` and re-`osc co` your packages individually.
  - **Run `osc co <prj> <pkg>` from the *parent* directory (the one that will contain `<prj>/`), not from inside an existing `<prj>/` checkout** — doing it from inside creates a nested `<prj>/<prj>/<pkg>`. (Real case: re-checking-out hardware packages from inside `hardware/` produced `hardware/hardware/fwts`.)
  - **A half-finished checkout that `osc up` greets with "Please run 'osc repairwc .'" is usually fastest fixed by `rm -rf <pkg>` + a fresh `osc co <prj> <pkg>`, not by repairing in place.** `osc repairwc` followed by `osc up` can re-hit `resuming broken update… PackageFileConflict: failed to add file '<f>' … already exists` and stay stuck; for an obs_scm/link package with no local edits, deleting and re-fetching is clean. (Real case: several `hardware` packages left mid-update by an interrupted checkout — `rm -rf` + `osc co` recovered each.)
- **`pgrep -f` / `pkill -f "<pattern>"` self-matches the very command running it**, because the pattern string is in that process's own argv — so it reports a phantom "still running" or kills its own shell (exit 144). To check for a real process, match the binary path with the bracket trick (`ps -eo pid,cmd | grep '[/]usr/bin/osc'`) or exclude self (`pgrep -f pat | grep -v $$`); never `pkill -f` a literal you also typed on the command line.

## Unattended / remote-build mode (the sanctioned exception to "build locally by default")

The two hard rules above assume an interactive session where a single local `osc build` is the fastest path to a verdict. **When the build/test block runs *unattended* (an autonomous agent run, a `/loop`, a multi-package cone where serial local builds would take hours), invert the default: drive the build through OBS remotely instead.** Local `osc build` is serial (one VM at a time) and blocks the session; OBS builds everything in parallel and re-triggers dependents automatically. The flow:

1. **HARD RULE: branch the package into your home project first — *always*, including packages you maintain — never build/commit in the devel project directly.** Holding maintainer rights on the devel package is **not** a licence to iterate in place: branch it anyway, so every remote build and commit happens on your throwaway copy and the shared devel project only ever sees the finished, green result via an SR. `osc branch <devel-project> <pkg>` creates `home:<you>:branches:<devel-project>/<pkg>` (or branch into a dedicated scratch `home:<you>:<topic>` for a whole cone — `osc meta prj -e` it with an `openSUSE_Factory` repo, path `openSUSE:Factory/standard`, arch `x86_64`). For a brand-new package there's nothing to branch — create it in the home project (see the new-package flow in `references/submit-watch.md`). Branching keeps all the iteration off the shared devel project until it's proven good. (The only time you commit straight into a project is the dedicated scratch `home:` project you created for the cone — that *is* your branch.)
2. **Apply the change in the branch, `osc commit`, and let OBS build it.** Skip the local `osc build` for the bulk verification; the server is now the build. (You may still do a quick local build of one stubborn package to iterate fast, but the cone as a whole builds remotely.)
3. **Monitor with `scripts/cone-status.sh <home-prj> [repo] [arch]`** — it prints a `code → package` table for the whole project in one call and sets a loopable exit code (`0` all green, `1` still in flight / dirty, `2` a *settled* failure), so you can `until scripts/cone-status.sh …; do sleep 30; done` instead of hand-parsing `osc results`. It deliberately won't report a failure while anything is still building or the result is `dirty` — encoding the stale-`F`-while-rebuilding lesson (a `failed` code on a package whose `_status` is `dirty="true"`/`scheduled` is a pending rebuild, not a real failure). Then **read `osc rbl <home-prj> <pkg> <repo> <arch>` for *every* package — the failed ones to fix, and the green ones for sub-threshold rpmlint findings / missed deps** (see the "read the successful builds' logs too" rule below). Manual decode if you need it: `.`=ok, `F`=failed, `U`=unresolvable (dep not published yet — transient right after committing its dependency), `B`=broken (often "no source uploaded" = staged but never `osc ci`'d).
4. **Gate the SR on the whole branch being green** — every package, every arch the repo enables (incl. `i586`), every `:test`/multibuild flavor — **plus** `source_validator`. Only then submit to the devel project.
5. **Submission (SR home→devel, accept, explicit Factory forward — there is no `--forward` flag) is Block 3** — see `references/submit-watch.md` "Accept-and-forward"; hand the green branch to the orchestrator.

This mode is *opt-in for unattended runs*; in a normal interactive session the local-build default still applies.

### Packaging a multi-package stack (a new dep cone) — build in a home: project, in parallel

When a new package drags in a chain of unpackaged deps (e.g. a Python app needing 10+ new modules), **do not build them one-at-a-time with local `osc build`** — local builds are serial (one VM at a time) and you'll wait hours. Instead create a scratch project (`osc meta prj -e home:<user>:<stack>`, add an `openSUSE_Factory` repo with path `openSUSE:Factory/standard`, arch `x86_64`) and **commit each prepped package into it**; OBS schedules them all in parallel and re-triggers dependents automatically as each leaf publishes. Workflow:
- **Determine build order leaves-first** but you don't have to wait between commits — commit everything you can, and the ones whose deps aren't published yet sit at `U` (unresolvable) and auto-rebuild the moment the dep appears. A transient `U` right after committing a dep is normal (publish lag), not an error — only worry if it persists after the dep shows `.` succeeded.
- **Monitor with `scripts/cone-status.sh <prj> [repo] [arch]`** (a `code → package` table + a loopable exit code: `0` all green, `1` in flight/dirty, `2` settled failure — wraps the `osc results` parsing and the stale-failure guard). Or read `osc results <prj>` directly (one column per package, dependency-sorted) — for the full status-code vocabulary see the canonical table in `references/submit-watch.md` ("`osc results` status vocabulary"); the short decode is in step 3 above.
- **Read a remote build log with `osc rbl <prj> <pkg> openSUSE_Factory x86_64`** (remote-buildlog; `osc buildlog` is the same for the current dir). Grep it for `error|fail|nothing provides|unpackaged|Unknown build hook`. A `B`-broken package has **no** log (`HTTP 404 … has no logfile`) — check `osc api /build/<prj>/<repo>/<arch>/<pkg>/_status` and `osc ls <prj> <pkg>` instead.
- Keep `osc st` for *local* working-copy state (uncommitted `A`/`M` files). Be aggressive about firing off independent `osc rbl` / `osc results` / fetches in parallel rather than serially.
- Only fall back to a local `osc build` when you need to iterate fast on one stubborn package in isolation; otherwise let the project build.
- **Read the `osc rbl` log of the *successful* packages too, not only the failed ones** — in a server-built cone a `.` (succeeded) only means rpmlint badness stayed under the 999 threshold and the build completed; it does not mean the log is clean. Grep each green package's log for the `RPMLINT report` section and any `W:`/`E:` lines, for sub-threshold rpmlint findings worth fixing (e.g. `devel-file-in-non-devel-package`, `non-executable-script`, weak-dependency suggestions), for compiler warnings that signal real bugs (`-Wmaybe-uninitialized`, `-Wimplicit-function-declaration`), and for **missed/incorrectly-auto-satisfied dependencies** (a runtime import that resolved only because a build dep happened to pull it in transitively). This is the server-side analogue of the "always read rpmlint after every build" hard rule — when you never ran the build locally, the remote log is the only place those problems show, so review it for every package in the cone before submitting. Filter the environmental noise this build farm always emits (the `polkit-default-privs … scriptlet failed`, `Disabling truststore because of missing certificates`, and `cycle: …` lines are not your package's problem).
- **Tightly inter-pinned clusters (e.g. the langchain/langgraph 1.x stack) need a coherent version *matrix*, chosen up front.** Each member pins the others to narrow ranges (langgraph 1.2.6 → langgraph-sdk `>=0.4.2,<0.5` + langchain-core `>=1.4.7` + langgraph-checkpoint `>=4.1.0,<5` + langgraph-prebuilt `>=1.1.0,<1.2`). Packaging a leaf at bare "latest" can pick a version the *target consumer* won't accept — instead resolve to ONE consumer version (the newest that satisfies your top-level need) and read its `requires_dist` to derive every member's exact version *before* building, or you'll rebuild leaves when constraints clash. (Real case: skillspector's cone — langgraph-sdk first built 0.3.15 for langgraph 1.0.10, then rebuilt to 0.4.2 once the matrix settled on langgraph 1.2.6.)
- **If you do iterate with local `osc build` on several cone packages at once, give each its own `--root=/var/tmp/build-root/<uniq>`** — concurrent local builds share the default buildroot and corrupt each other mid-init (`.init_b_cache` rpms vanishing). The server-side home:-project parallel build above sidesteps this entirely; the per-`--root` trick is only for the occasional local iteration.
- When the whole cone is green, SR each package to `devel:languages:python` (leaves first so deps exist in the target). If the user wants to maintain them, add the user to each package `_meta` (`<person userid="…" role="maintainer"/>`) **and** say so in the SR message.

### Deployment cone in `home:pluskalm` — links to devel, host-installable repo

`home:pluskalm` is NOT a scratch area: it is the **curated deployment project** for tools the user runs from RPM on the host (the MCP servers: `skillspector`/`-mcp` + `bugzilla-mcp` and their dependency cones). Rules:

- **Every package is an `osc linkpac` `_link` to the devel project where it is maintained** (devel:tools, devel:languages:python, science:machinelearning, science:machinelearning:mcp, …) — never a copypac/branched copy. Links auto-track devel updates, so the deploy repo stays current without manual bumps.
- **Repos:** `16.0` (path `openSUSE:Leap:16.0/standard`, aarch64 — the host's install repo, added to zypper from `https://download.opensuse.org/repositories/home:/pluskalm/16.0/`) and `openSUSE_Tumbleweed` (x86_64+aarch64 canary).
- **Filling the Leap dependency gap:** after linking the top packages, iterate the `16.0` repo's `unresolvable` states — for each `nothing provides <binary>`, map the binary to its Factory source package, find its devel project (`osc develproject`), and `linkpac` that too. Do NOT link anything Leap 16.0 already provides (an unresolvable on a package Leap *does* have = version-floor conflict → real blocker, report it). Repeat until zero unresolvable, watch to green with `cone-status.sh home:pluskalm 16.0 aarch64`.
- **Transient/experimental work goes to `home:pluskalm:scratch`** (or another `home:pluskalm:<topic>` subproject) — never accumulate one-off packages in `home:pluskalm` itself. Scratch subprojects are throwaway; the deploy project is stable.

## Updating a package that uses a source service (`_service` / tar_scm)

Some packages carry a `_service` (and a `_servicedata`) instead of a hand-managed tarball — the source archive is regenerated by OBS source services (`tar_scm`/`obs_scm`, `set_version`, `recompress`, …), usually in `mode="disabled"` (they run only when invoked locally, not on every server commit). For these, **a version bump is a `_service` edit + a service run, not a manual tarball swap**:

1. **Bump the revision in `_service`.** Change the `<param name="revision">` (e.g. `v0.9.5` → `v0.9.15`). That tag/branch/commit is what gets exported. (Always verify the new tag is genuinely newer — see "Checking for newer upstream versions"; tag *dates*, not string order.)
2. **Run the services locally:** `osc service runall` runs every service in `_service` regardless of `mode` (this is what executes `disabled` services). It re-clones/fetches, exports the new revision to a fresh tarball, `set_version` rewrites `Version:` in the spec, `recompress` produces the configured compression, and `_servicedata`'s `changesrevision` advances to the new commit.
3. **Verify it actually moved.** `tar_scm` can print a benign `merge: origin/<tag> - not something we can merge / Already up to date` and still be wrong if a stale cache wasn't refreshed — so confirm: `_servicedata` `changesrevision` changed, the new tarball's top-dir is `<name>-<newversion>/`, and the version string inside the sources matches. Then `osc rm` the old tarball and `osc add` the new one.
4. **Remove the leftover clone cache.** `osc service runall` of a `tar_scm` git service leaves a working git checkout dir (`<pkg>/`) in the package directory. `osc status` does **not** list it (it's untracked and osc-ignored), so it won't be committed — but `rm -rf <pkg>` it anyway to avoid confusion and stray disk use.
5. **`changesgenerate` produces a raw git-log dump — always curate it.** When `_service` has `<param name="changesgenerate">enable`, the run prepends a `.changes` entry containing **every git commit subject** between the old and new `changesrevision`. Across a multi-version jump that is 50+ one-line bullets — which violates the changelog guideline ("summarise for an openSUSE *user*, ~10–15 lines, don't dump upstream's log"). Keep the service-stamped date/author header, then **replace the bullet dump with a curated user-facing summary** (new default behaviour, new platform/toolchain support, notable fixes) before committing. Real case: kcbench 0.9.5→0.9.15 generated ~60 bullets; collapsed to ~14 lines.

After the service run, the rest is normal: `spec-cleaner`, local `osc build`, `osc service runall source_validator`, then commit. On commit, the `_service`/`_servicedata` changes and the new tarball go up together; the disabled services do **not** re-run server-side.

**A rolling `nightly` (or other non-release) tag breaks `@PARENT_TAG@` versioning — restrict tag matching with `match-tag`.** For a service that tracks a *branch* (`<param name="revision">main</param>`) with `<param name="versionformat">@PARENT_TAG@+@TAG_OFFSET@</param>`, `git describe` picks the **most recent tag reachable from HEAD**. If upstream adds a moving tag *on* the HEAD commit (e.g. a `nightly` tag that CI re-points every build), `@PARENT_TAG@` resolves to `nightly` and `@TAG_OFFSET@` to `0`, producing a bogus version like `nightly+0` — which is a **downgrade** versus the previous `1.0.1+196` (rpm vercmp ranks the digit-leading string newer than the alpha-leading one) and breaks the update. Fix by telling the service to consider only release tags: add **`<param name="match-tag">v*</param>`** (it passes `--match 'v*'` to `git describe`), so the version goes back to `1.0.1+<offset>`. Also confirms why a snapshot's parent tag may be an *older* release than the latest tag — the higher `vX` tag simply isn't on the tracked branch's ancestry. (Real case: neovim-gtk — upstream's new `nightly` tag on `main` HEAD produced `nightly+0`; `match-tag=v*` restored `1.0.1+279`.)

### Tagless upstream (no tags, no releases, not on PyPI) — pinned-commit `obs_scm`

Some upstreams never cut a release in any form: zero git tags, zero forge releases, nothing on PyPI (real case: NVIDIA/skillspector). Two rules:

- **Never ship a locally-built sdist/tarball whose version exists nowhere upstream.** factory-auto runs a download check against every `Source:` URL; a `files.pythonhosted.org` (or any) URL that 404s is an automatic decline ("Source URLs are not valid"), and a hand-built tarball has no verifiable provenance even if the URL were served. (Real decline: skillspector "2.3.10" — a locally-built sdist one patch-version past upstream's static 2.3.9, submitted with a phantom PyPI URL.)
- **The fix is a pinned-commit `obs_scm` snapshot** — `scripts/scm-snapshot.sh <pkg> <url> <sha>` scaffolds and verifies it: version `X.Y.Z~gitYYYYMMDD.<hash8>` where `X.Y.Z` is upstream's own declared version (pyproject/CMake), committed `.obscpio` so nothing needs downloading at review time. Two traps it already encodes, for when you hand-write the `_service`: `%h` in `versionformat` abbreviates to **7** chars — pin an explicit `<param name="version">` if you want 8; and verify the produced `.obsinfo` version string before committing.
- **Python + static pyproject version → the dist-info keeps the RELEASE version, not your snapshot `%{version}`.** hatchling/setuptools emit `<name>-<pyproject-version>.dist-info` (e.g. `skillspector-2.3.9.dist-info`) while the RPM version is `2.3.9~git…`, so `%{python_sitelib}/<name>-%{version}.dist-info` in `%files` no longer matches and the build dies on unpackaged files. Hardcode the upstream version's dist-info path with a comment flagging it for the next bump.

### Prefer a real release tarball over the service when one exists

**When a `_service` tracks a git *tag* and upstream publishes that same released tag as a tarball, prefer switching to the upstream tarball directly and dropping the whole service machinery.** A real release tarball is simpler, verifiable, and avoids the git-clone/repack churn — the service adds no value once a proper released tarball exists for the same revision. This commonly comes up when a package was put on a git-snapshot scheme (`X.Y.Z+git.%cd`) during a period when upstream wasn't tagging, and upstream has since resumed cutting releases.

Mechanics:
1. Set `Source0:` to the upstream tarball URL (full HTTPS), and update `Version:` to the release.
2. `osc rm _service _servicedata *.obscpio *.obsinfo` (drop the whole service set), then `osc add <name>-<version>.tar.*`.
3. Build, validate, commit as usual.

**Caveats:**
- This applies only to a *released* tarball matching the tracked tag. If the package **deliberately tracks unreleased git snapshots ahead of the last tag** (a documented reason to be ahead of upstream's releases), keep the service — see "Checking for newer upstream versions". Confirm the snapshot scheme isn't intentional before ripping it out.
- A snapshot→release switch can change the **version string** and (for libraries) often coincides with a **soname bump** — recheck `%define sover` against the new `version-info`/`SONAME` and update the lib subpackage name *and* `baselibs.conf` (see "Shared libraries"). Re-check `%files` too: a release can add/rename/move installed files (a new binary, a relocated header) that a snapshot didn't, surfacing as `Installed (but unpackaged) file(s) found` at the end of the build.

### After switching to a tarball, recheck whether `autoconf`/`automake` are still needed

For an **autotools** package, whether you still need `autoconf`/`automake`/`libtool` + an `autoreconf` step depends on **which kind of tarball** you switched to — verify, don't assume either way:

- A **GitHub/GitLab git auto-archive** (`/archive/refs/tags/<tag>.tar.gz`) is a raw git snapshot: it ships only `configure.ac`/`Makefile.am` (and often `autogen.sh`), **no** generated `configure`/`Makefile.in`/`aclocal.m4`. So `autoreconf` *is* required → keep `autoconf`/`automake`/`libtool`. (When no release *asset* exists for the tag — `gh api …/releases/tags/<tag>` 404s, or the assets list is empty — the auto-archive is your only option.)
- A **real `make dist` release tarball** (a maintainer-uploaded release asset) ships a pre-generated `./configure` — so `autoreconf` and the autoconf/automake/libtool BuildRequires can be **dropped**.

**Check:** extract the tarball and `ls configure Makefile.in aclocal.m4`. `configure` present → drop the regen tooling; absent (only `configure.ac`/`autogen.sh`) → keep it. (Real case: libiscsi 1.20.3 — no release asset, so the tag archive lacked `configure`; autoconf/automake/libtool and the `autoreconf -fiv` were kept. nbd 3.27.1 — same situation, autoreconf added in `%build`.) Run the regen in whichever phase the spec already uses (`%prep` is typical; `%build` before `%configure` is also fine). Note that this distinction also means **the SR's `factory-auto` download check still verifies the auto-archive URL** — that's fetchable on a standard GitHub port, so it passes.

## Common build pitfalls

When a build fails locally, check this list before debugging from scratch.

- **Parallel local `osc build`s of DIFFERENT packages MUST use dedicated `--root=` build roots.** `osc build` defaults every aarch64 build to the same `/var/tmp/build-root/standard-aarch64`; two concurrent builds (e.g. two agents updating two packages) silently share it and the second init clobbers the first mid-build. Symptoms in the victim are misleading: mass test failures on one flavor, a "vanished" source tree, files owned by the wrong package — looking exactly like a real regression in the change under test. Give each concurrent build its own root (`osc build --root=/var/tmp/build-root/<pkg> …`) and re-run any clobbered build with `--clean` in a dedicated root before believing its failures. (Real case: a zathura build wiped a concurrent python-langsmith `%check` mid-run; py313 "failed" en masse while py311/py314 had already passed — the rerun in a dedicated root was 100% green.)
- **The buildroot exports `XDG_*` variables — tests that set a custom `$HOME` get poisoned.** `aaa_base` ships `/usr/etc/security/pam_env.conf.d/xdg.conf`, so every PAM login in the chroot (`su - abuild`, i.e. the build itself) exports `XDG_CONFIG_HOME=/home/abuild/.config` etc. — in **every** SUSE buildroot (Factory and SLE/Leap alike). A testsuite that sets `HOME=/home/test` and expects XDG default paths derived from it fails, because glib (and most XDG implementations) prefer a non-empty `XDG_CONFIG_HOME` over `$HOME`. Symptom: an assertion comparing `/home/<custom>/.config` against `/home/abuild/.config`. Fix: run the test with the `XDG_*` variables unset (`env -uXDG_CONFIG_HOME -uXDG_DATA_HOME … ./test`), preferably as an upstream(able) patch. (Real case: girara's `utils-with-home` — upstream commit `0f653f2c4e3e`, backported as `girara-tests-unset-xdg-vars.patch`.)
- **A `%check` skip keyed on `meson test --list` output is meson-version-fragile — it can silently stop skipping (or silently skip forever).** `meson test --list` prints `project:testname` on meson ≥ 1.11 but bare `testname` on older meson, so a `${test_list//project:name}` substitution no-ops on the other format. Both failure modes are bad: on one codestream the "skipped" test suddenly runs and FTBFS (girara on Leap 16.1, meson 1.8.1); on the other the test is skipped forever and nobody notices the suite isn't actually running (girara on Factory). Prefer *fixing* the test; if you genuinely must skip, use an explicit mechanism (`%meson_test --no-suite <s>`, or pass explicit test names) — never string-substitution on list output.

- **HARD RULE: prove a build/flag fix is actually *effective* before claiming or submitting it — a clean build does not mean your change did anything.** A compiler flag or `-D` you add can be silently overridden, leaving a byte-identical binary; you then ship a no-op (and, worse, a `.changes` entry that lies about fixing a bug). Two traps and their checks:
  - **Last `-D`/flag wins.** If the upstream build already sets the macro you're trying to set, *order on the command line decides*, and the build's own flags usually come **after** your `XCFLAGS`/`%optflags`. Grep the real compile line in the build log (`grep -- '-Dyour_macro' …/.build.log | grep -oE '-Dyour_macro=[^ ]*'`) — if your value isn't the last occurrence, it's dead. (Real case: **mupdf** — added `-DLCMS2MT_PREFIX=mupdf_lcms_` to `XCFLAGS` to namespace the bundled lcms2, but `LCMS2_CFLAGS` appends `-DLCMS2MT_PREFIX=lcms2mt_` later on *every* line, so the prefix was unchanged.)
  - **Diff the produced binary against the *current distro* binary.** Fetch the published RPM (`osc api /build/<prj>/<repo>/<arch>/<pkg>/<file>.rpm > x.rpm`) and compare what actually matters — exported/undefined symbols (`nm -D`), `DT_NEEDED` (`objdump -p`), file list. If your rebuild is equivalent, the change is inert and the real cause is elsewhere. (Real case: mupdf again — the rebuilt `libmupdf.so` was identical to Factory's (437 `lcms2mt_` symbols, no undefined `cms*`, no `liblcms2` in NEEDED); the actual zathura-crash root cause — a *static*-link symbol interposition against system lcms2 — had already been fixed by a prior static→shared conversion, so the bug needed **no mupdf change at all**, only verification and a bug close.)
- **HARD RULE: extract the upstream changes FIRST — it is the opening step of a version bump, not a closing one.** Immediately after confirming the new version (and *before* editing the spec), pull the changelog / release notes / commit range (`gh api repos/<o>/<r>/compare/<oldtag>...<newtag> --jq '.commits[].commit.message'`, the release body, `NEWS`/`ChangeLog`/`CHANGES` in the tarball, or a diff of the shipped files for data packages). Do this once, up front, and let it drive the whole update: the same extracted change-set is what you (a) act on as the spec-affecting checklist below, and (b) curate into the `.changes` entry at the end. Figuring out the changes only at `.changes`-writing time means you've already done the build-fail-fix-rebuild loop blind and likely missed a workaround you could have dropped or a dep you could have anticipated. Front-loading turns "discover problems one failed build at a time" into "know what's coming."
- **HARD RULE: read the upstream changelog/release notes as a checklist of *spec-affecting* changes — on every version bump, ask "does anything here change how our spec should look?"** The changelog is not just `.changes` fodder; it routinely announces things that should drive a spec edit, and a workaround left in place after upstream fixed the underlying problem is dead weight. Scan each release's notes for items in these categories and act on them:
  - *"bumped cmake_minimum_required / require CMake X", "switched to meson/cmake", "now needs C++17"* → a `-DCMAKE_POLICY_VERSION_MINIMUM=`, `-std=`, build-system, or compiler workaround in the spec may now be droppable (or newly needed). (Real case: rtags 2.44 bumped `cmake_minimum_required` to ≥3.5 across its CMakeLists, so the spec's `-DCMAKE_POLICY_VERSION_MINIMUM=3.5` could be dropped — verified by a clean rebuild that then only emits a deprecation warning.)
  - *"added/removed/renamed dependency", "vendored X removed", "new backend"* → adjust `BuildRequires` (see next bullet).
  - *"added RISC-V/arch support", "dropped Windows"* → update `ExclusiveArch`/`ExcludeArch` (see below).
  - *"fixed build with <compiler/lib> N"* → a downstream patch for exactly that is probably now obsolete (test it — it usually no longer applies / reverse-applies).
  - *"new test framework", "tests now require X"* → re-evaluate `%check` (see `%check` rules).
  - *"removed tool/feature Y"* → drop its `--enable`, deps, and `%files` entry.
  Treat every spec workaround (a `-D…` flag, an `export CFLAGS=-std=…`, a `sed`, a patch, a `-j1`, a `||:`) as a hypothesis that a new release may have invalidated — re-test and remove what's no longer earning its place. Dropping a now-unnecessary workaround warrants a brief `.changes` bullet tying it to the upstream change.
- **A version bump frequently adds new build dependencies — re-resolve them, don't assume the old `BuildRequires` set still suffices.** New releases pull in libraries the old one didn't, and `configure`/`cmake` fails one missing dep at a time, so expect to iterate: build → read the *first* `configure: error: <X> not found` / `Libtool library used but 'LIBTOOL' is undefined` / `error: nothing provides …` → add the matching `BuildRequires` → rebuild → repeat. Map each error to the openSUSE provider (`pkgconfig(libxxhash)` ← `xxhash-devel`, `pkgconfig(liburcu)`, `pkgconfig(libzstd)`, `libtool` for a new libtool library, etc.) and prefer the upstream-stated minimum version. Reading the new `configure.ac`/`CMakeLists.txt` up front (grep `PKG_CHECK_MODULES`/`find_package`/`AC_CHECK_LIB`) lets you front-load the additions instead of N build cycles. (Real case: partclone 0.3.37→0.3.47 newly required libtool, libxxhash, liburcu, libzstd.)
- **When you go to *enable a new optional feature/driver* and its dep is *transiently* uninstallable in the target (a soname mismatch mid-rebuild, not a permanent gap), keep the feature wired but `%bcond_with`-disabled with a comment — don't rip it out.** A dep can be momentarily broken because *it* hasn't been rebuilt against a just-bumped lower-level library: e.g. `apache-arrow` still requiring `libprotobuf.so.<old>` after Factory bumped protobuf → `osc build` dies at dep-resolution `nothing provides libprotobuf.so.34.1.0 needed by libarrow_flight…`. That's not your package's fault and not a reason to drop the driver. Leave the `BuildRequires`/`-D…=ON` behind a `%bcond_with <feat>` (default off) plus a comment naming the blocker and the re-enable (flip to `%bcond_without`), so a one-liner restores it once the dep is rebuilt — and **don't** advertise the disabled feature in `.changes`. Distinguish from a *permanent* missing dep (needs onboarding) by checking whether the dep exists at all vs. just being uninstallable right now. (Real case: gdal — Arrow/Parquet kept as a disabled `%bcond_with arrow_support` while AVIF/CFITSIO/etc. shipped; submitted the rest, deferred Arrow.)
- **A GitHub/GitLab auto-archive (`/archive/refs/tags/…`) ships only what's committed — *generated* artifacts (man pages, `configure`, …) are absent and must be regenerated at build time.** Beyond the `configure`/autoreconf case (see "Check archive type"), watch for **man pages generated from DocBook/asciidoc/help2man**: a `docs/Makefile.am` with `man_MANS = foo.8` built by a `%.8 : %.xml` rule running `xsltproc --nonet $(DOCBOOK_MAN_STYLESHEET)`. The auto-archive has the `.xml` but not the `.8`, so you must add `BuildRequires: libxslt-tools` (provides `xsltproc`) + `docbook-xsl-stylesheets` (provides the local stylesheet + XML catalog for offline `--nonet` resolution). **The failure is sneaky**: the make rule is often written `-@($(XSLTPROC) …)` (leading `-@` = ignore-errors), so when the tool is absent the man pages just silently don't get built, and the only symptom is `install: cannot stat './foo.8'` at `%install` time — not an obvious "xsltproc not found". (Real case: partclone 0.3.47.)
- **When a release's changelog/notes mention *new architecture* support, update `ExclusiveArch`/`ExcludeArch` to match.** A spec restricted to `ExclusiveArch: x86_64 aarch64` will silently keep *not* building newly-supported arches until the list is widened. If the upstream changelog says e.g. "added RISC-V support", add `riscv64` (and reconsider any `ExcludeArch`). You often can't build a foreign arch locally (qemu buildconfig pins the *host* — riscv64 needs an x86_64 host; see Local builds), so this is a legitimate case to enable the arch and let the **OBS server-side build** validate it — watch that arch's result and back it out if it FTBFS. Conversely, if a release *drops* an arch, narrow the list. (Real case: xmrig 6.26.0 added a RISC-V JIT → `ExclusiveArch: x86_64 aarch64` widened to add `riscv64`.)
- **A new release can *remove/disable* a tool or feature — drop the matching `--enable`, `BuildRequires`, and `%files` entry.** Read `Makefile.am`'s `*_PROGRAMS=` lines for commented-out targets. (Real case: partclone 0.3.47 commented out `partclone.imgfuse` in `sbin_PROGRAMS`, so `--enable-fuse` built nothing yet `configure` still demanded FUSE3 — the fix was to drop `--enable-fuse`, `pkgconfig(fuse)`, and the `partclone.imgfuse` `%files` line entirely, not to add `pkgconfig(fuse3)`.) Conversely a new release can *add* binaries/man pages → `%files` then fails with `Installed (but unpackaged) file(s) found` (add them) or a renamed file fails with `File not found` (e.g. partclone's bash completion `partclone-prompt`→`partclone-completion`, mtd-utils' moved `README`). Re-check `%files` against the produced RPM (`rpm -qlp`) on every bump.
- **Treat `||:` / `|| true` / a make-rule leading `-` as suspect on every bump — they mask real failures; try removing them and rebuild clean.** A `%make_build … ||:` (often left behind with a stale "this error seems harmless" comment) swallows a non-zero make exit, which can hide a genuinely broken build or, e.g., leave artifacts unbuilt. On a version bump, drop the masker (and any now-stale hardcoded `LIBS=`/`CFLAGS=` overrides that a cleaner upstream Makefile no longer needs) and rebuild: if it's green, the hack was obsolete and is gone for good; if it now fails, you've surfaced a real issue to fix properly instead of shipping past it. In general, remove such error-suppression rather than carrying it forward. (Real case: partclone 0.3.47 — dropping `LIBS="…" ||:` built cleanly.)
- **On a download failure, check whether upstream *moved hosting* before assuming a transient blip.** Dead project FTP/host is common for older projects; the tarball may have relocated. (Real case: mtd-utils' `ftp.infradead.org` now 403s — releases moved to `https://infraroot.at/pub/mtd/`; updating `Source0`/`Source1` to the live host fixed both the local fetch and factory-auto's download check.) For a **GPG-signed** tarball, verify the new release's `.asc` against the package keyring: `gpgv --keyring` *fails on an ASCII-armored keyring* (`invalid packet (ctb=2d)`), so import into a throwaway homedir instead — `GNUPGHOME=$(mktemp -d) gpg --import <pkg>.keyring && gpg --verify <tarball>.asc <tarball>` → expect `Good signature`; if the signing key isn't in the keyring, upstream rotated keys and the keyring needs updating.
  - **When the upstream site is *gone* (HTTP 410 / dead domain) but a later release exists, a distribution often mirrors the pristine upstream tarball — source it from there.** Debian's pool keeps the unmodified `*.orig.tar.*` for every version it packaged; if Repology/another distro shows a newer version your dead upstream can't serve, fetch Debian's orig tarball (`https://deb.debian.org/debian/pool/main/<l>/<src>/<src>_<ver>.orig.tar.<ext>`) and point `Source0` at it with a `#/<name>-<version>.tar.<ext>` rename and a comment noting the original site is gone. `deb.debian.org` is a standard-port, persistent host, so factory-auto's download check passes. (Real case: w_scan — `wirbel.htpc-forum.de` 410 Gone; the final `20210218` release — a gcc-10-compile-fix repackage of `20170107` — was mirrored as `w-scan_20210218.orig.tar.bz2` in Debian and sourced from there.) Distinguish from the `w_scan2` *fork* (different project, semver `1.0.x`) — verify the mirror's tarball is the *same* project by its ChangeLog/contents before adopting.
- **Audit `%configure` flags against the upstream's actual `configure --help`** — when modernising or reviewing a spec, don't assume existing flags are load-bearing because a past `.changes` entry mentions them. Extract `configure` (or `configure.ac`) from the source tarball and check each flag's documented effect. Common offenders to drop on sight: `--with-largefile` (default on modern systems, often removed from upstream autoconf and prints `configure: WARNING: unrecognized options`), `--enable-dependency-tracking` (per `configure --help`: *"do not reject slow dependency extractors"* — a leniency knob for ancient compilers, no-op on modern gcc; the opposite `--disable-dependency-tracking` is what distro builds actually want for a small speedup). A wrong-sounding past `.changes` entry like "enable more thorough dependency tracking" is a strong tell that the original packager misread the flag.
- **Fortran program SIGSEGVs at startup (exit 139)** — almost always a stack overflow caused by large PROGRAM-scope or SUBROUTINE-scope local arrays. Fortran allocates these on the stack by default; the Linux default stack limit is 8 MB. Confirm by running the binary under `ulimit -s unlimited`: if it then completes, the diagnosis is stack overflow. Persistent fix: add `-fmax-stack-var-size=0` to the gfortran compile command so all large locals go to BSS/heap; this bakes the fix into the installed binary instead of relying on each user lifting `ulimit`. Document the flag with a comment in `%build` so future maintainers don't strip it. Real case: `stream.f` declares 3 × 2M double arrays (~48 MB) at PROGRAM scope and SIGSEGVs on aarch64; `-fmax-stack-var-size=0` fixes it.
- **GENERAL RULE — re-inspect the *whole* BuildRequires list on every update for continued relevance, and treat a major version bump as a mandatory full re-audit.** A spec's BR set drifts out of truth over time: deps get added for features later removed, renamed upstream, absorbed into another package, or made transitive — and new mandatory deps appear. On *every* version bump don't just bump `Version:`; walk each `BuildRequires` and ask "does the new source still use this?", and separately "did the new source start needing something not listed?". Cheap mechanics: `grep -r <lib/header/pkgconfig-name> src/ CMakeLists.txt cmake/ meson.build` for each suspect (no hits → candidate to drop); read what the build system actually wires (`find_package`/`pkg_check_modules`/`include(<dep>)` for CMake, `dependency()` for meson, `go.sum`/`Cargo.lock` for vendored langs); and let the **build itself be the oracle** — the offline build fails fast on a *missing* dep (add it back) and a clean build with a dep removed proves it was dead. The biggest wins come from a package that **stops building a bundled component**: every BR that existed only for that bundled build becomes dead in one stroke (e.g. switching a consumer from a statically-bundled library to the system shared one drops the whole bundled-build dep cluster, keeping only the consumer's *own* direct deps + the new `-devel`). Order of magnitude: this is where most stale-BR cruft hides, so spend the most audit effort on major jumps and bundling changes. The language-specific and feature-removal cases below are special cases of this rule.
- **Audit BuildRequires after an upstream language rewrite (C → Rust/Go).** When upstream rewrites a project in a new language, the old C-era `*-devel` BRs almost always linger in the spec long after the binary stops linking them. Detection is cheap and worth doing on every major version bump that crosses a language boundary: for each suspect `*-devel`, (1) `grep -r <libname> src/ Cargo.lock` (or `go.sum` / `vendor/`) — no hits is a strong signal; (2) inspect the built binary in the build root with `chroot /var/tmp/build-root/<repo>-<arch> ldd /usr/sbin/<binary>` — only the libraries actually listed there need a corresponding `-devel` BR (modulo bindgen-style build-time-only deps like `clang-devel`). Real case: thin-provisioning-tools 1.x kept `BuildRequires: device-mapper-devel` from its C ancestor even though the Rust port didn't link libdevmapper — verified by both grep and ldd, dropped cleanly. Don't blindly remove a BR that *might* be a build-time tool (bindgen wants clang headers, some crates shell out to `pkg-config`-discovered libs at build time); the grep+ldd combo is what makes the removal safe. **But the audit is per-version, not once-and-done — a dropped dep can come back.** The *same* thin-provisioning-tools later (1.3.x) **re-introduced** the libdevmapper need when a new `migrate` feature pulled in the `devicemapper-sys` crate (an FFI `-sys` crate that links libdevmapper); the BR had to return (as `pkgconfig(devmapper)`). Tell-tales that a Rust dep needs a C `-devel` again: a `*-sys` crate in `Cargo.lock` (`grep -A1 'name = ".*-sys"'`), `clang-devel` already in the BRs (bindgen), and `links = "<lib>"`/`pkg_config` in the vendored crate's `Cargo.toml`/`build.rs`. So **re-grep `Cargo.lock` for `-sys` crates on every update** — never trust a prior "it doesn't need X" verdict, and never drop a BR on memory alone.
- **A version bump that *removes an upstream feature* leaves the matching BuildRequires dead — watch the release notes/changelog for "removed X" lines.** When the curated changelog or upstream notes say a feature/subsystem was dropped, check whether a BuildRequires existed only for it and drop it. Detect by grepping the new source for the library/usage (`grep -rni <lib> src/ CMakeLists.txt`) — a "removed" warning string with no remaining link usage confirms the BR is dead. Real case: monero 0.18.5.0 removed built-in UPnP ("UPnP port mapping support was removed"), so `libminiupnpc-devel` was a dead BR and was dropped. (spec-cleaner won't catch this — it's a semantic change, not a style one.)
- **A major version bump of old C code often breaks on Factory's current GCC — expect it, it is not your spec's fault.** GCC 14 turned `-Wimplicit-function-declaration`, `-Wincompatible-pointer-types` and `-Wint-conversion` into **errors** by default; GCC 15 defaults to **C23** (`-std=gnu23`), where an empty parameter list `T (*fp)()` means *zero* args (not "unspecified"), so a call `fp(arg)` fails with `too many arguments to function 'fp'`. These are hard errors, not warnings — they are not caused by `-Werror` and `-Wno-error=` will not silence the C23 one. Prefer a **targeted source patch** when there are few sites: `grep -rnE '\(\s*\*\s*[A-Za-z_]+\s*\)\s*\(\s*\)' src/*.c` counts empty-paren function-pointer declarations (ior 4.0.0 had exactly one — patched `void(*fp)()` → `void(*fp)(void*)`). `-std=gnu17` (set in `CFLAGS`; `export CFLAGS="%{optflags} -std=gnu17"` before `%cmake`) is a blunt whole-package fallback that restores pre-C23 semantics — appropriate for old K&R C (e.g. an inline `double atof();`) where patching every site is excessive. **Check upstream `main` first** — the fix is almost always already there but unreleased: `gh api repos/<o>/<r>/contents/<path>?ref=main --jq .content | base64 -d` shows the current file, so you can backport the exact upstream change and tag it `PATCH-FIX-UPSTREAM`.
  - **C23 also promotes `bool`/`true`/`false`/`nullptr`/`static_assert`/`thread_local` to keywords**, so old code that does `typedef _Bool bool;` (or `typedef unsigned char bool;`) now fails with `'bool' cannot be defined via 'typedef'`. Same `-std=gnu17` fix.
  - **A package can FTBFS on *some arches only* because upstream forces the C standard for just one arch.** Watch for a CMake/configure branch like `if(CMAKE_SYSTEM_PROCESSOR MATCHES "x86_64|amd64") … set(CMAKE_C_FLAGS "… -std=gnu99") …`: x86_64 then gets a pre-C23 standard and builds, while aarch64/ppc64le/s390x inherit GCC 15's C23 default and break — so Factory shows the package green on x86_64 but **failed on every other arch**. Fix by forcing the standard for *all* arches (`export CFLAGS="%{optflags} -std=gnu17"` before `%cmake`). (Real case: blosc 1.21.6 — `bool` typedef in `shuffle.c`, only non-x86 failed.)
  - **An i586-only `%check` failure on an exact float==double comparison is x87 excess precision — and the two "obvious" fixes are a no-op and a footgun respectively.** A test that stores a `float` literal into a `double` (or otherwise round-trips float↔double) and then asserts *exact* equality passes on x86_64/aarch64/… but fails **only on i586**, because GCC defaults to x87 (`-mfpmath=387`, 80-bit) on 32-bit x86: the stored/reloaded value comes back as the true double instead of the float-widened value (e.g. `0.12345` vs `0.12345000356435776`). Two tempting fixes are wrong: (a) **`-fexcess-precision=standard` is usually a no-op** here — it governs *expression* excess precision, not the x87 store/convert, so it builds but doesn't change the result (verify per the "prove a flag fix is effective" HARD RULE above — diff the failing test output, don't assume). (b) **`-mfpmath=sse -msse2` is unsafe**: the openSUSE i586 baseline has **no SSE2** (`Optflags: i586` carries no `-march`, so it's the `i586-suse-linux`/Pentium default; the separate `i686` line is also pre-SSE2), so forcing SSE2 makes the library **SIGILL on the very hardware the i586 port targets**. The correct fix is to make the *test* representation-robust — **narrow both sides back to `float`** (cast, or the project's `asFloat()`/`toFloat()` accessor) so they collapse to the same value on every arch; this keeps the round-trip check and is upstreamable. **Verifying an i586 fix from a non-x86 host:** local `osc build … i586` runs under QEMU user emulation and is slow/flaky (package-cache `hdrmd5` mismatch loops); instead commit to the devel project and watch the **server-side** i586 result, confirming the specific test now passes, *before* the Factory SR. (Real case: jsoncpp 1.9.8 — `ValueTest/objects` compared `0.12345f` exactly against the stored double; patched to `numericFound->asFloat()`, server i586 went green, submitted upstream as gh#open-source-parsers/jsoncpp#1700.)
  - **The C++ analogue: a dependency's headers may newly *require* C++17, while the project's own code needs an older standard — and C++17 library names can clash with the project's identifiers.** Current **googletest** headers `#error` on `< C++17`; building the test suite then forces C++17 on code that may use constructs C++17 removed or names it now defines. Set `CMAKE_CXX_STANDARD 17` (or `-std=gnu++17`) for the whole package, and fix clashes by qualifying: under `using namespace std;` + `using namespace itpp;`, a call to the project's `any(...)` became ambiguous with C++17's `std::any` → qualify as `itpp::any(...)`. (Real case: itpp — re-introduced to Factory after this fixed the FTBFS that got it bot-deleted.)
- **A big version jump can cross an upstream *build-system migration* — budget for rewriting `%build`/`%check`/`%install` and the whole `BuildRequires` set, not just bumping `Version`.** The common case is **Autotools → CMake**: replace `autoreconf`/`%configure`/`%make_build` with `%cmake`/`%cmake_build` (add `BuildRequires: cmake` + `ninja` and `%define __builder ninja`), `make check` → `%ctest`, `%make_install` → `%cmake_install`, and **drop `autoconf`/`automake`/`libtool`**. Map the old `--with-foo`/`--enable-foo` switches to the new `-DWITH_FOO=`/`-DBUILD_FOO=`/`-DENABLE_FOO=` options — read them from `option()`/`cmake_dependent_option()` in the top `CMakeLists.txt` (don't guess names). The same jump often **removes whole features** (and thus their deps) — read the release notes / `doc/dependencies.md`: e.g. bitcoin 28→31 dropped BerkeleyDB (legacy wallet), miniupnpc (UPnP), protobuf+openssl (BIP70), so `libdb-4_8-devel`/`libminiupnpc-devel`/`pkgconfig(protobuf)`/`pkgconfig(openssl)` all became dead BRs. A **Qt5 → Qt6** migration rides along similarly: take the required modules from the upstream `find_package(Qt6 …COMPONENTS…)` list, express them as `pkgconfig(Qt6Core)` … (with the upstream min version, e.g. `>= 6.2`); `lrelease`/`Qt6LinguistTools` comes from a *separate* `qt6-linguist-devel` (no pkgconfig form, like its qt5 sibling); and **rename the GUI subpackage** (`foo-qt5` → `foo-qt6`) with **`Obsoletes: foo-qt5 < %{version}-%{release}` only — do NOT add a matching `Provides: foo-qt5`.** The `Obsoletes` carries the upgrade (the new package replaces the installed `foo-qt5`), but a `Provides: foo-qt5` would be a *lie*: the Qt6 build does not provide the "qt5" capability, and falsely advertising it could wrongly satisfy something that genuinely wants the qt5 GUI. (Real case: a Factory reviewer flagged exactly this on the bitcoin 31.0 SR — *"that is likely a lie… the qt5 capability is not available; only keep the obsoletes"*.) This is the general rule for a **flavor/toolkit rename**: `Provides` of the old name is only honest when the new package is a true drop-in providing the same capability; for a Qt5→Qt6 (or similar) change it isn't, so `Obsoletes`-only. Disable optional new subsystems that drag in heavy new deps unless wanted (bitcoin's `-DENABLE_IPC=OFF` avoids Cap'n Proto).
- **For a hard repackaging problem (build-system migration, install-layout change, soname-scheme change), consult how OTHER distros already package it before inventing a solution.** Pull the recipe from a fast-moving distro: Arch `PKGBUILD` (`https://gitlab.archlinux.org/archlinux/packaging/packages/<pkg>/-/raw/main/PKGBUILD`), AUR (`https://aur.archlinux.org/cgit/aur.git/plain/PKGBUILD?h=<pkg>`), Alpine `APKBUILD`, Fedora (`https://src.fedoraproject.org/rpms/<pkg>/raw/rawhide/f/<pkg>.spec`), Debian. They've usually already solved the same wall, and their `prepare()`/`build()`/`package()` shows the canonical patch or install-rearrange. (Real case: vapoursynth R74+ switched to a **wheel-only meson build** that installs everything into Python `site-packages`; Arch's PKGBUILD showed the approach — build the wheel and symlink/redirect the libs/headers/pkgconfig into FHS, with the proper upstream SONAME `libvapoursynth.so.4`.) **Corollary:** if the package is *not* packaged by any of them (e.g. it's normally vendored into a larger project), then **upstream's own files are authoritative** — follow upstream's `*.service`/install layout/defaults rather than guessing. (Real case: nqptp is bundled into shairport-sync everywhere, not packaged standalone — so upstream's `DynamicUser` service file was the reference, doubly trustworthy because the same author maintains both ends.)
- **A *freshly released* upstream can still FTBFS against a bleeding-edge openSUSE library — and the fix is usually already merged upstream post-release, so backport it.** Factory tracks libraries very aggressively (e.g. Boost 1.91 the same month it lands), so even a brand-new upstream tag may predate the compatibility fix. Don't assume "latest upstream = builds": when it breaks on a system lib, search upstream for the fix before hand-rolling one — `gh api -X GET search/commits -f q='repo:<o>/<r> <lib> <keyword>'` / `search/issues … type:pr`, fetch the commit as a `.patch` (`curl -sL https://github.com/<o>/<r>/commit/<sha>.patch`), add it `PATCH-FIX-UPSTREAM` and note the PR number. Real case (separate from the const-qualifier one below): bitcoin 31.0 failed to compile against **Boost ≥ 1.91** (Boost rewrote `multi_index` on mp11, rejecting the "index list as a derived `struct … : indexed_by<…>`" pattern → `static assertion failed: is_index_list<…>` with no other error); upstream PR #35175 (merged after the 31.0 tag) inlines `indexed_by<>` directly — backported verbatim as `fix-boost-1.91-multi_index.patch`.
- **GCC 15 C++ FTBFS: `-Werror=discarded-qualifiers` from now-`const`-preserving `strchr`/`memchr`/`strstr`/`strpbrk`.** glibc's string/search functions now keep the `const` of their argument, so `char *p = strchr(const_str, …)` / `char *e = memchr(const_buf, …)` fails with *"assignment discards 'const' qualifier from pointer target type"*. Fix **per site by intent, not blanket**: if the pointer is only read (arithmetic, comparison) make it **`const char *`**; if the code legitimately **writes through it** (e.g. `*p = '\0'`), the underlying buffer must be mutable (verify — often a VLA/`strdup` copy with overly-conservative `const` typing) and an explicit **`(char *)` cast** is correct. These are independent of `-Werror`/`-Wno-error=` won't help cleanly; the fixes are upstreamable. To enumerate every site at once, build with the project's werror relaxed (`-Dwerror=false` for meson, `-Wno-error=discarded-qualifiers` in CFLAGS) and grep the log. (Real case: yambar — `tag.c` cast (writes), `modules/script.c` const (reads) → dnkl/yambar#497.)
  - **Exception — when the warning comes from the package's *own* hardcoded `-Werror` AND upstream is archived/dead, neutralise the `-Werror` instead of patching each warning.** Some old projects bake `-Werror` into their `AM_CFLAGS`/`CMAKE_C_FLAGS`, so *every* new GCC warning is a fatal FTBFS, and a dead upstream will never accept fixes — you end up carrying an ever-growing pile of "fix GCC N warning" patches (and the package keeps getting bot-deleted between bumps). The durable rescue is to drop upstream's blanket `-Werror` while **keeping the distro's `-Werror=return-type`**: pass **`make … CFLAGS="-Wno-error %{optflags}"`** (or `%cmake -DCMAKE_C_FLAGS="-Wno-error %{optflags}"`). Ordering is the trick — the compiler sees `AM_CFLAGS(-Wall -Werror) … CFLAGS(-Wno-error … %optflags-with-Werror=return-type)`: the `-Wno-error` cancels upstream's earlier `-Werror`, and `%{optflags}`'s `-Werror=return-type` (which comes *after* the `-Wno-error` in the line) re-enables the one gate the distro insists on. Escape the `%` if you mention `%%{optflags}` in a comment (rpmlint `macro-in-comment`). (Real case: **simple-obfs** — archived upstream already carried hand-cherry-picked GCC8 + GCC9 `-Werror` patches; GCC15 `-Wdiscarded-qualifiers` triggered a `[botdel]`. Rather than add a third per-GCC patch, `CFLAGS="-Wno-error %{optflags}"` fixed it durably and rescued it from deletion. Contrast yambar above — *live* upstream, so per-site patches + a PR were right.)
- **A dependency's *headers* can raise the required C++ standard or need a now-missing include — the consumer FTBFS even though its own code is unchanged.** (a) When a library bumps its public headers to use `std::optional` / `std::is_convertible_v` / `std::string_view` etc. (e.g. **UHD 4.x** is C++17), a consumer pinned to an older `CMAKE_CXX_STANDARD` fails to *compile the dependency's headers* (`'optional' in namespace 'std' does not name a template type`). Fix: bump the consumer's standard (`set(CMAKE_CXX_STANDARD 17)` / `-std=c++17`). (b) **Newer Boost dropped transitive includes** — `boost::lexical_cast` (and friends) were previously pulled in by another boost header and now must be `#include <boost/lexical_cast.hpp>`'d explicitly (`'lexical_cast' is not a member of 'boost'`). Add the include to *every* `.cpp` that uses it (grep `boost::lexical_cast`). These are two **independent** root causes → two patches + two PRs (see the one-issue-one-patch rule). (Real case: soapy-uhd — `cxx17.patch` (SoapyUHD#75) + `boost-lexical-cast.patch` (SoapyUHD#76).)
- **`%autosetup` for a patch that touches a *subdirectory* file: pass `-p1` explicitly.** A bare `%autosetup` can apply a root-level hunk but then skip a `modules/foo.c` hunk with `patch: **** No file to patch. Skipping patch. / 1 out of 1 hunk ignored` and fail `%prep`, even though the patch is `-p1`-clean by hand. `%autosetup -p1` fixes it. Make `-p1` the default habit for any package whose patches reach into subdirs. (Real case: yambar's `modules/script.c` hunk.)
- **Don't assume CMake installs every binary to `%{_bindir}` — verify the install destinations, especially for test/aux binaries.** A project's custom install logic can route some targets elsewhere: bitcoin's `install_binary_component(... INTERNAL)` macro installs the *internal* binaries (`test_bitcoin`, `bench_bitcoin`, `test_bitcoin-qt`) to **`${CMAKE_INSTALL_LIBEXECDIR}`** (`/usr/libexec`), while user-facing tools go to `bin` — so a `%files` listing them under `%{_bindir}` fails packaging. Read the `install(TARGETS …)` / install-macro definitions, or just enumerate the produced RPM's file list (`rpm -qlp …`) after the build, and point `%files` at the real paths. (test/bench binaries that upstream *doesn't* install at all must instead be installed by hand in `%install` or the subpackage dropped.)
- **Upstream `cmake_minimum_required(VERSION <X>)` with X < 3.5 is a hard error under CMake 4 — fix it with `-DCMAKE_POLICY_VERSION_MINIMUM=3.5`.** CMake 4.0 *removed* compatibility with `cmake_minimum_required` below 3.5: `CMake Error: Compatibility with CMake < 3.5 has been removed from CMake`. Passing **`-DCMAKE_POLICY_VERSION_MINIMUM=3.5`** to `%cmake` makes such trees build again without patching every (often bundled) `CMakeLists.txt`. (Real cases: bitcoin, monero-gui, diskscan all needed it.) Note the *threshold*: a declared minimum **≥ 3.5 but < 3.10** only emits a deprecation *warning* ("compatibility … will be removed from a future version") and still builds — so add the flag when you see the **hard error** (or pre-emptively when upstream declares `< 3.5` and you want to future-proof), not for every deprecation warning.
- **A CMake package that FTBFS with bizarre module errors (`__check_type_size_impl ... incorrect arguments`, `FindPackageHandleStandardArgs.cmake:NNN (include): could not find requested file`) is usually shipping *stale bundled copies of standard CMake modules* that shadow the system ones.** Old projects add their source tree to `CMAKE_MODULE_PATH` and bundle their own `CMakeParseArguments.cmake`, `FindPackageHandleStandardArgs.cmake`, `FindPackageMessage.cmake`, `Find<lib>.cmake`, … which take precedence over `/usr/share/cmake/Modules` and break against modern CMake (e.g. a bundled `CMakeParseArguments` overrides the built-in command and corrupts `check_type_size`'s arg parsing). **Detect:** for each `*.cmake` the package bundles, check whether a same-named file exists under the system `…/cmake/Modules` — those are the shadows. **Fix:** delete the bundled copies in `%prep` so CMake uses its own (`for m in CMakeParseArguments FindPackageHandleStandardArgs FindPackageMessage Find…; do find . -name "$m.cmake" -delete; done`). **Caveat:** some bundled `Find<lib>` modules are *customised* and set variables the project's `CMakeLists` consumes by a non-standard name — after dropping them, the system finder may set a different var (e.g. ossim used `${TIFF_LIBRARY}` (singular, only its bundled finder set it) while system `FindTIFF` sets `${TIFF_LIBRARIES}`), causing **link** failures (`undefined reference to TIFF*`). Fix those with a small patch switching to the system finder's variable. Real case: ossim 2.12.0 — dropping all 10 bundled standard modules + one `TIFF_LIBRARY`→`TIFF_LIBRARIES` patch + `-std=gnu17` made it build on current CMake/GCC (it had been *removed from Factory* for this FTBFS).
- **A release tarball can ship an optional backend/module that simply does not compile** — upstream's CI never built that configuration. ior 4.0.0 shipped `src/aiori-RADOS.c` still written against the *old* AIORI API (references struct members that no longer exist, `errno` undeclared, incompatible function-pointer inits) while the rest of 4.0.0 had migrated; it could not compile at all (not a flags problem). When it is one optional backend and upstream `main` has already fixed it but not released, the pragmatic distro fix is to **disable that backend** — drop its `--with-X`, its `*-devel` BuildRequires, and any `%ifarch`/`ExclusiveArch` that existed only for it — rather than backport a large rewrite. Document it as temporary in `.changes` ("disabled: broken in this release, fixed upstream, will return next release") so the next maintainer re-enables it. Confirm the backend is the *only* failing translation unit first: `grep 'error:' <buildlog> | grep -v <thatfile> | sort -u` — if anything else fails, that needs its own fix.
- **`binary-or-shlib-defines-rpath` is a badness-10000 rpmlint error that fails the *server* build, even though local `osc build` still wrote the RPMs.** A bundled test framework's link flags (e.g. `-Lcheck/lib64 -Wl,-rpath=check/lib64`) routinely leak a **relative** RUNPATH into the installed binaries. The asymmetry matters: local `osc build` produces the `.rpm`s and *then* reports the rpmlint error, so "RPMs were written" is **not** proof of a clean build — always read the final `N errors, … N badness` rpmlint line (badness over the threshold = the OBS build will abort). Fix: `BuildRequires: chrpath` + `chrpath -d %{buildroot}%{_bindir}/<bins>` in `%install`. A relative rpath cannot resolve at runtime anyway, so deleting it is safe.
- **A `meson`/`meson-python` package with `wrap-git` subprojects fetches dependency sources from git at configure time — which FAILS in the offline OBS build.** A `subprojects/*.wrap` file containing `[wrap-git]` (`url = …git`, `revision = …`) makes meson `git clone` the dependency during `meson setup`; OBS has no network, so the build dies. Detect: `dependency('foo', …)` in `meson.build` paired with `subprojects/foo.wrap`. **Fix: bundle the wrap'd sources as Source tarballs and place them where meson expects them, so it uses the local copy instead of cloning.** Add each subproject's release tarball as a `SourceN:`, then in `%prep` extract + rename into `subprojects/<wrap-basename>/` (the directory name must match the `.wrap` filename minus `.wrap`): `tar -xf %{SOURCE1} -C subprojects && mv subprojects/<upstream-topdir> subprojects/<wrapname>`. meson then resolves the dependency from the local subproject (which provides it via `meson.override_dependency('<name>', …)`). When the dep is `static: true`, it's statically linked into the extension, so no separate shlib package is needed and bundling is self-contained. (Real case: `python-siphash24` 1.8 — its meson build wrap-git'd `libcsiphash-1` (c-util/c-siphash) which itself wrap'd `libcstdaux-1` (c-util/c-stdaux); both were added as Source tarballs, extracted into `subprojects/libcsiphash-1` / `subprojects/libcstdaux-1`, and built offline.)
- **A CMake project that pulls deps via `FetchContent`/`ExternalProject` with `GIT_REPOSITORY`/`URL` clones them at configure time — FAILS offline. Patch each module to use the system copy.** Detect: `cmake/modules/*.cmake` (or inline `CMakeLists.txt`) with `FetchContent_Declare(... GIT_REPOSITORY ...)`. For each, replace the fetch with the system equivalent: `find_package(<Pkg>)` (target name usually matches, e.g. `fmt::fmt`), or `pkg_check_modules(... IMPORTED_TARGET ...)`, or set the include-dir variable to `${CMAKE_INSTALL_PREFIX}/include` for a header-only dep. When the project hard-codes a target name (e.g. links `ReflexLibStatic`), create an `INTERFACE` library of that name aliasing the system target so you don't have to touch the consuming `CMakeLists`. Generate the changes as a real `PatchN:` (edit copies + `diff -u`). **Companion Ninja gotcha:** an `ExternalProject_Add(... BUILD_BYPRODUCTS libfoo.a ...)` with a *relative* byproduct path resolves it against the **top build tree**, but if the project references the artifact by a **source-tree** absolute path (common with `BUILD_IN_SOURCE 1`), Ninja errors `'…/libfoo.a', needed by '…', missing and no known rule to make it` — fix by patching `BUILD_BYPRODUCTS` to the same **absolute** path the project consumes. (Real case: falcosecurity-container-plugin — fmt/re-flex/plugin-sdk-cpp/btrfs FetchContent → system, plus the go-worker `libworker.a` byproduct made absolute.)
- **Ship a *shared* library, never a static one, unless absolutely necessary — and if a dependency ships *no linkable library at all*, fixing that upstream-distro package is the right move.** openSUSE policy strongly prefers `.so`s in a versioned `libNAME<sover>` subpackage. A package can silently ship **nothing linkable**: its autotools build produces only static `.a`s which `%install` then deletes (`find … -name '*.a' -delete`), leaving a dangling `.pc` — so any consumer's `-lfoo` fails with `cannot find -lfoo`. The clean fix is to build it the way that yields a real shared lib: **many such projects also ship a `CMakeLists.txt` that builds a `SHARED` lib + a CMake package config** even when the default autotools path doesn't — switch the spec to `%cmake`/`%cmake_build`/`%cmake_install`, patch the CMake to add `VERSION`/`SOVERSION` and drop the static target, split out a `libNAME<sover>` subpackage, and (since the CMake build may install neither) re-add the man page / `.pc` by hand. Confirm nothing else regresses first (`osc whatdependson` + a reverse build-dep search); if the lib currently doesn't exist, *nothing* links it, so the switch is safe. (Real case: re-flex 6.0.0 — autotools shipped no lib at all; converted to its upstream CMake build to ship `libreflex.so.6` + `Reflex::` cmake config, in a new `libreflex6` subpackage.)
- **Don't recompress a `_service`-fetched source tarball to `.tar.zst` just for size — the minimal build chroot may lack `/usr/bin/zstd`** (`%prep` dies with `sh: /usr/bin/zstd: No such file or directory`). Either keep the `tar` service output as a plain `.tar` (drop the `recompress` service — simplest, and a single `osc service manualrun` then produces everything unattended) or recompress to `.gz` (universally present). Reserve `.zst` for cases where the package already pulls `zstd` in, or add `BuildRequires: zstd` deliberately.
- **A brand-new package whose spec has an empty `%changelog` fails the build with `error: %source_date_epoch_from_changelog is set, but %changelog has no entries to take a date from`.** rpm derives `SOURCE_DATE_EPOCH` from the newest `.changes` entry; a new package with no `.changes` (or an empty one) has nothing to read. **Fix: create the `<name>.changes` with an initial entry before the first build** — a single `- Initial package (version X.Y)` bullet under the standard dated/author header (see "Adding a .changes entry"). This is the new-package analogue of the always-add-a-changelog rule.

### Functionally test the built package when a change warrants it

"It built" is not "it works." For **API/ABI ports** (fuse2→fuse3, openssl 1.1→3, a Qt major bump), **behaviour changes**, or anything where a clean compile doesn't prove the feature still functions, install the freshly-built RPM into a throwaway **podman container of the target distro** and exercise the real behaviour — better to verify than to ship a broken package. (Skip it for pure changelog/metadata changes.)

```
podman run --rm --device /dev/fuse --cap-add SYS_ADMIN \
  --security-opt apparmor=unconfined --security-opt label=disable \
  -v "$DIR:/rpm:ro" registry.opensuse.org/opensuse/tumbleweed bash /rpm/test.sh
```
- FUSE/mount tests need `--device /dev/fuse --cap-add SYS_ADMIN`; `--security-opt label=disable` avoids the SELinux *"Permission denied"* on the bind-mount; `chmod -R a+rX` the shared dir first (rootless userns maps your files).
- **Assert the actual function, not just that it starts.** (Real case: the tarix fuse2→fuse3 port — created a tar, `tarix -i` an index, `fuse_tarix` mounted it, then `cat`/`readlink` the mounted files **and** confirmed writes still return EROFS — i.e. that the read path works *and* the retained read-only stubs still reject writes.)

### Switching a package off a dead upstream fork to a maintained lineage

When the package tracks a **dead fork** and the CVE/bug fix only exists in a *different* upstream lineage that the other distros moved to (the cross-distro survey reveals this — see `bugzilla-cve-triage.md`), the fix is a **lineage switch**, not a patch. Mechanics that recur (real case: **xar** — mackyle 1.6.1, dead since 2012 → Apple `apple-oss-distributions/xar` build 503, as Debian/Fedora/Gentoo all did):

- **Reuse a distro's patch set + build recipe for the new lineage** rather than inventing one — Gentoo's `files/` (or Debian's `debian/patches/`) for the new upstream are the Linux-build adaptation you need. Mirror the distro that's closest to your target version.
- **The new source often builds from a sub-directory** and ships generated artifacts: e.g. Apple xar builds in `xar/` with `autogen.sh`. Apply the distro patches into that subdir (`%patch -P N -p1 -d <subdir>`), and replicate the distro's `src_prepare` fix-ups (header moves, `__APPLE__`→`__NO_APPLE__` seds, RPATH strip) in `%prep`.
- **Regenerate `configure` when a patch touches `configure.ac`** (the shipped `configure` is then stale): run `./autogen.sh --noconfigure` / `autoconf`. Check `autogen.sh` to see whether it needs `automake`/`libtool` (Apple xar uses hand-written Makefiles — `autoconf` alone).
- **Target the newest build, but expect a portability gap the distro patches don't cover** — they're written for an *older* version, so newer code has un-guarded platform-isms. (xar 503's new `xar_fdopen_digest_verify()` used macOS-only `F_GETPATH`; fix on Linux with a `/proc/self/fd/<fd>` `readlink` fallback under `#ifdef F_GETPATH … #else …`.) Find these by building and reading the first hard `: error:` (filter out the warnings).
- **Confirm the lib didn't change ABI:** check the soname major (`LIB_REV`/`*.so.N`) so the `lib<name>N` subpackage name and `baselibs.conf` stay correct; and re-verify the **license** and the `%doc`/`%license` file *names* (a different lineage may drop `NEWS`, rename `COPYING`, etc.).
- **Functionally round-trip the result** (e.g. create+extract an archive) — a lineage switch is exactly the kind of change where "it built" isn't enough.

## New package from scratch

**When creating a BRAND-NEW package, survey other distros for the *packaging itself* before writing a line of spec — HARD RULE.** This is distinct from the fix-survey hard rule (Core directive item 8, which surveys for *fixes* when touching an existing package): here you are surveying for the *packaging structure* of something Factory does not have yet. Before authoring the spec, look at how Fedora, Debian, Gentoo, Arch, Alpine, openEuler, Void, NixOS, FreeBSD ports, OpenMandriva and Mageia already package it (`scripts/distro-survey.sh <pkg>` covers all of them in one call, plus read their actual `.spec`/`debian/rules`/`PKGBUILD`/ebuild/Void-template/Nix-derivation), **and** check for an existing `home:` project copy on OBS (`osc search --package <pkg>`).

Harvest from them: the **shared-library soname / subpackage split**, the `-devel`/`-tools`/`-doc` partitioning, build-system invocation and **build options** (`-D…`/`--enable-…`), `%check`/test wiring, any **required patches**, and the correct **SPDX license**.

Prefer basing the new package on the best existing reference — `osc copypac` from a good `home:` copy (see the copypac note), or port a distro's spec — then reconcile it against the cross-distro consensus, rather than writing from scratch and missing a subpackage split or a needed patch everyone else already carries. Treat a single distro's choices as one data point; where distros **disagree** (e.g. one ships only a static lib, another a versioned `.so`), follow openSUSE's Shared Library Policy and the majority of the distros that ship a proper shared lib. (Real case: cmark-gfm — new to Factory for mupdf's Markdown support; surveyed Fedora/Debian/Arch/etc. to settle the `libcmark-gfm`/`libcmark-gfm-extensions` soname subpackages + `-devel` split before packaging it in `Publishing`.)
