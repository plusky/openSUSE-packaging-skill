# Language-specific packaging quick reference

When packaging language-specific software, fetch the matching wiki page for the full rules — these summaries cover only the structural choices.

- **Vendored-dependency packages (Go / Rust / Node / any language whose deps are fetched at build time): generate the vendor bundle with a `_service`, the thin-provisioning-tools way — do NOT hand-vendor, and do NOT download a language toolchain yourself to run `go mod vendor` / `cargo vendor` by hand.** The canonical setup is a `_service` file (services `mode="manual"`) that fetches the source and produces the vendor tarball, whose output you then commit as *tracked sources* and consume from the spec. Worked model (thin-provisioning-tools, a Rust pkg — the same shape applies to Go with `cargo_vendor`→`go_modules`):
  ```xml
  <services>
    <service name="obs_scm" mode="manual">      <!-- fetch upstream at the tag -->
      <param name="url">https://github.com/…/foo.git</param>
      <param name="scm">git</param>
      <param name="revision">refs/tags/v1.3.2</param>
      <param name="versionformat">@PARENT_TAG@</param>
      <param name="versionrewrite-pattern">v(.*)</param>
      <param name="changesgenerate">enable</param>
    </service>
    <service name="tar" mode="manual"/>
    <service name="recompress" mode="manual"><param name="file">*.tar</param><param name="compression">zst</param></service>
    <service name="set_version" mode="manual"/>
    <service name="cargo_vendor" mode="manual">   <!-- Go: name="go_modules" -->
      <param name="srcdir">foo</param>           <!-- Go: point at the dir holding go.mod -->
      <param name="compression">zst</param>
      <param name="update">true</param>
    </service>
  </services>
  ```
  Then in the spec: `Source0: %{name}-%{version}.tar.zst`, `Source1: vendor.tar.zst`, and `%autosetup -a1` (or `-p1 -a1`) to unpack the vendored tree next to the source. Commit the generated `%{name}-%{version}.tar.zst` **and** `vendor.tar.zst` (plus `_service` / `_servicedata` / `*.obsinfo`) as real package files. Run the services with `osc service manualrun` (the host needs the matching `obs-service-cargo` / `obs-service-go_modules` and toolchain) **or** let them run server-side — either way the point is that the *service* produces `vendor.tar.*`, never a by-hand vendor dir. This is the preferred way to create any new vendored-deps package; the per-language rows below give the build macros.
- **Require a compiler/runtime by *versioned virtual provide*, not by pinning the implementation package.** `BuildRequires: go >= 1.25` (the `go1.N` packages each `Provides: go = 1.N`, so this floats forward to 1.26+) — never `BuildRequires: go1.25`, which nails it to one compiler package. Same idea for other versioned virtuals (`java >= 1.x`, etc.).
- **Packaging a Falco / libsinsp *plugin* (a `lib<name>.so` loaded by libsinsp): the plugin links the header-only plugin SDK only — NOT libsinsp.** A plugin is `dlopen`'d by the inspector at runtime, so its build needs `plugin-sdk-cpp` (+ whatever else its own code links), and it picks up the *system* libsinsp at load time. The trap is that the upstream build often pulls a **from-source libsinsp** in only for its **unit tests** (a `sinsp_test_support` target) — so building with the tests **off** (`-DENABLE_TESTS=OFF` or equivalent) drops the entire nested-libsinsp build and lets the plugin build against the shared stack. Install the `.so` under `%{_libdir}/falcosecurity/plugins/`. (Real case: falcosecurity-container-plugin — `include(libs)` lived only in `test/CMakeLists.txt`; tests off → no libsinsp needed, links just plugin-sdk-cpp + fmt + shared re-flex + the Go worker.)

### Rust (cargo) deep-dive (beyond the table row)

All learned packaging grok-build (a ~1270-crate Cargo workspace) — they generalize to any large Rust app:

- **Audit the repo for network-fetching build hooks BEFORE the first build** — an offline OBS build dies on any of these, each with a confusing late error:
  - `rust-toolchain.toml` pins an exact toolchain and makes the build try to invoke **rustup**: `rm -f rust-toolchain.toml` in `%prep` and build with the distro rust (`BuildRequires: rust >= <pin>` keeps the floor honest; Factory's rust is usually newer).
  - **dotslash launchers** — small JSON-manifest files in `bin/` (e.g. `bin/protoc`) that download the real binary at first use. `rm -f` them in `%prep` and point the build at the system tool (`export PROTOC=%{_bindir}/protoc`, `BuildRequires: pkgconfig(protobuf)`).
  - **build.rs / xtask scripts that download prebuilt helper binaries** (a bundled ripgrep, node, etc.). Look for an env-var escape hatch in the build script and point it at the system binary (e.g. `export GROK_TOOLS_BUNDLE_RG_PATH=%{_bindir}/rg` + `BuildRequires: ripgrep`); if the tool is also invoked at runtime, add a matching `Requires:`.
- **Big-workspace OOM: `%cargo_build` parallelism is keyed to cores, not RAM** (`-j$RPM_BUILD_NCPUS`), and a single heavy crate's rustc can peak ~7 GB RSS — on a many-core/low-RAM worker the kernel OOM-kills it. Signature: build `failed` **nondeterministically per worker/arch**, log shows `Out of memory: Killed process NNN (rustc) total-vm:…` followed by `error: could not compile <crate>`. Fix with BOTH levers: a `_constraints` file (`<hardware><physicalmemory><size unit="G">16</size></physicalmemory></hardware>`) so the scheduler picks big-RAM workers, **and** `BuildRequires: memory-constraints` + `%limit_build -m 2000` at the top of `%build` so the job count is capped by available memory regardless of core count.
- **`unstripped-binary-or-object` on a Rust binary is almost never a spec bug — it's a debuginfo-disabled project.** `%cargo_build` deliberately keeps DWARF (`-C debuginfo=2 -C strip=none`) so `find-debuginfo.sh` can split it into `-debuginfo`/`-debugsource` packages and strip the binary — but that machinery only runs where the project meta has `<debuginfo><enable/>` (devel projects and Factory do; `home:` projects by default do NOT, so local/home builds warn). Check `osc meta prj <prj> | grep -A2 debuginfo` before "fixing" anything. **No Factory Rust package does manual `strip`, `%global debug_package %{nil}`, or `CARGO_PROFILE_RELEASE_STRIP`** (verified across ripgrep/bat/helix/gitui/wasmtime/alacritty/eza/starship/uv/fd/bottom) — manual stripping is an anti-pattern that destroys the debuginfo package. Exception to watch: if upstream's *default* release profile sets `strip = true|"debuginfo"`, DWARF dies at link time and the debuginfo package would be empty — only then neutralize it (sed the profile line, or `CARGO_PROFILE_RELEASE_STRIP=none`).
- **Tagless upstreams (repo synced from a monorepo, no releases): pin a commit, don't track a branch.** `obs_scm` with `<param name="revision">FULL_SHA</param>`, `<param name="versionformat">0+git%cd.%h</param>`, `changesgenerate` disabled (hand-write the entry). When the `tar` + `recompress` services produce the `.tar.zst`, the intermediate `.obscpio`/`.obsinfo` need not be committed — track only `_service`, the `.tar.zst`, `vendor.tar.zst`, spec, changes, patches.
- **`ExclusiveArch: %{rust_tier1_arches}` — use it bare.** The macro is always defined in real openSUSE build environments; don't wrap it in an `%if "x%{?rust_tier1_arches}" == "x"` fallback (an x86_64-only fallback would silently drop aarch64 in the env where it fired).
- **Distro-neutering patches for agent/updater-style apps:** an app that self-updates fights RPM (patch its installer-detection to report none), and default-on telemetry should be forced off for the distro build (patch the config resolver to return disabled before any config/env/remote override). Keep both patches surgical early-returns and name them in `.changes`.

### Python singlespec deep-dive (what `py2pack` won't get right)

`py2pack fetch <name> && py2pack generate <name>` scaffolds a spec but the result always needs hand-cleanup. The recurring gotchas, all learned building the FastMCP cone:
- **Pick the build backend's helper modules from `pyproject.toml`, not guesswork.** Read `[build-system] requires` *and* the `[tool.hatch.*]` hooks: `[tool.hatch.version] source = "vcs"` → add `BuildRequires: %{python_module hatch-vcs}` (the Factory package is `python-hatch_vcs` with an underscore, but `%{python_module hatch-vcs}` resolves via Provides); `[tool.hatch.metadata.hooks.fancy-pypi-readme]` → `%{python_module hatch-fancy-pypi-readme}`; `uv_build` backend → `%{python_module uv-build}`; `uv-dynamic-versioning` → `%{python_module uv-dynamic-versioning}`; poetry-core, setuptools-scm, etc. likewise. A missing build hook fails late as `hatchling.plugin.exceptions.UnknownPluginError: Unknown build hook: vcs` / `metadata-generation-failed`.
- **Derive runtime deps from the sdist's `PKG-INFO` `Requires-Dist`, not the README.** Base (unconditional) deps + whichever `extra ==` groups you actually need (e.g. a meta-package `foo[client,server]` → union the `client` and `server` extras). Put each in `%{python_module X}` `BuildRequires` (runtime deps must also be build deps or `%check`'s import fails with `ModuleNotFoundError`). `pydantic[email]` → also pull `email-validator`.
- **`W: python-leftover-require` on a `uv_build` / `uv-dynamic-versioning` package is a FALSE POSITIVE — do NOT remove the flagged `Requires:`; they are load-bearing.** The trap: openSUSE's automatic Python dep generator (`pythondistdeps`) reads runtime deps from the built wheel's `.dist-info/METADATA` `Requires-Dist`. For **hatchling / poetry-core / setuptools** backends this works, so a manual `Requires:` that *also* appears there is genuinely redundant — but for **uv-backend wheels (`uv_build`, and hatchling + the `uv-dynamic-versioning` metadata hook) the deps never make it into the wheel METADATA**, so the generator emits *nothing* and the package's only source of runtime deps is your hand-written `Requires:`. rpmlint, seeing a `Requires:` not backed by METADATA, mislabels each as `python-leftover-require` — but deleting them silently drops every runtime dep, which only surfaces later as a `ModuleNotFoundError` in a *dependent's* `%check` (e.g. a consumer that `BuildRequires` the package no longer pulls its transitive deps). **Decide by the built RPM, never by the warning:** `osc api '/build/<prj>/<repo>/<arch>/<pkg>/<pkgflavor>-<ver>.noarch.rpm?view=fileinfo_ext'` and look at the `python3NN-*` requires — if it lists only `python3NN-base` (no module deps), the generator is NOT covering this package, so the manual `Requires:` are mandatory; keep them and accept the cosmetic `leftover-require` warnings (or silence them with an `rpmlintrc`). Only when the auto-Requires already lists the modules is a manual `Requires:` truly redundant. (Real case — the mistake to NOT repeat: fastmcp-slim and py-key-value-aio (both uv-backend) were "cleaned" by deleting their manual `Requires:` to hush 63 and 12 leftover-require warnings; their RPMs then required only `python3NN-base`, and the next rebuild of `python-fastmcp` died at `%check` with `ModuleNotFoundError: No module named 'platformdirs'`. The deps had to be restored.)
- **Provide each runtime dep as both `%{python_module X}` `BuildRequires` and (for uv-backend packages) an explicit `python-X` `Requires`.** Build deps are needed for `%check`'s import; runtime `Requires` are needed because uv wheels hide them from the generator (above). For hatchling/poetry/setuptools packages the generator supplies the runtime `Requires` for you, so don't duplicate there.
- **Casing false-positives in Factory dep resolution** — try the canonical PyPI casing: `Authlib` (not authlib), `PyYAML` (not pyyaml), `typing_extensions` (underscore), `Pygments`. If `%{python_module x}` reports unresolvable but the package clearly exists, it's almost always casing.
- **`%check`: use a manual import smoke test, not `%pyproject_check_import`.** `%pyproject_check_import` can expand to a shell line that dies with `/var/tmp/rpm-tmp: line N: fg: no job control`, and full test suites drag in unpackaged pytest plugins (`pytest-asyncio`, `aiomisc-pytest`, `--cov`/addopts). Prefer `%python_expand PYTHONPATH=%{buildroot}%{$python_sitelib} $python -B -c "import MODULE"` (use `%{$python_sitearch}` for C extensions). Note the module's *import* name can differ from the dist name (src-layout: `uncalled_for`, namespace: `key_value.aio`).
- **The `python-bytecode-inconsistent-mtime` rpmlint error is systemic in this build env** (appears even without vendoring). **First triage which kind it is:** for a *new* package with no local release number it can be a pure local `SOURCE_DATE_EPOCH` artifact that's clean on the server — `osc build --release 1` confirms that (see update-build.md). But if it persists with a release set, or the package already exists and a *current* toolchain rebuild surfaces it (an old server build can predate the check), apply the permanent, SDE-independent fix in `%install` on every Python package: `%python_expand $python -m compileall -q -f -o 0 -o 1 --invalidation-mode unchecked-hash %{buildroot}%{$python_sitelib}/MODULE`. The `-o 0 -o 1` is required — without both, only the `.pyc` (not `.opt-1.pyc`) gets the hash-based invalidation and half the errors remain. Either hash mode works for rpmlint (`unchecked-hash` is the openSUSE default; `checked-hash` also clears it). Use `$python` (shell var), never `%{$python}` (expands literally to `%{python311}` → `fg: no job control`). (Real case: archmage — 36 errors on a current aarch64 rebuild though the months-old Factory x86_64 build was green; the in-spec recompile fixed it everywhere.)
- **Console scripts (a `[project.scripts]` entry → `/usr/bin/NAME`) need the singlespec alternatives dance**, else the build fails with `Installed (but unpackaged) file(s) found: /usr/bin/NAME`: in `%install` add `%python_clone -a %{buildroot}%{_bindir}/NAME`; add `%post`→`%python_install_alternative NAME`, `%postun`→`%python_uninstall_alternative NAME`; and `%python_alternative %{_bindir}/NAME` in `%files`.
- **A man page shipped through alternatives must be registered as a `--slave`, or its `/etc/alternatives/NAME.1.gz` symlink dangles.** If the package also `%python_clone -a`'s a man page and lists `%python_alternative %{_mandir}/man1/NAME.1%{?ext_man}` in `%files`, then **extra arguments to `%python_install_alternative` become `--slave` links** — so `%post`→`%python_install_alternative NAME NAME.1` (and the macro auto-appends `.gz`). With only `%python_install_alternative NAME`, the install scriptlet registers just the binary, the man-page symlink is never created by update-alternatives, and you get a dangling `/etc/alternatives/NAME.1.gz` (rpmlint won't always catch it). `%postun` stays `%python_uninstall_alternative NAME` (removing the main link drops its slaves). Confirm with `rpm -qp --scripts <rpm>` — the install line should end `… --slave /usr/share/man/man1/NAME.1.gz NAME.1.gz /usr/share/man/man1/NAME-3.NN.1.gz`. (Real case: archmage boo#1202977 — Leap 16.0's older python-rpm-macros support the slave arg too, verified by build.)
- **Verify the `%doc` filename against the tarball** (`tar tzf … | grep -i readme`) — `README.md` vs `README.rst` mismatch fails at packaging with `File not found: …/README.rst`. Same for `%license`: omit it entirely if the sdist ships no LICENSE/COPYING file (many `*-slim` redistribution packages don't).
- **C-extension modules** (e.g. caio): `BuildRequires: %{python_module devel}` + `gcc`, drop the bundled `*.c`/`*.h` the wheel ships alongside the `.so` (`find %{buildroot} -name '*.c' -delete; … '*.h' …`, else `devel-file-in-non-devel-package` badness), and use `%{python_sitearch}` (arch-specific) throughout instead of `%{python_sitelib}`.
- **Python-version floor → flavour selection.** `requires-python >= 3.13` → `%define pythons python313` (single flavour); a `< 3.11` conditional dep (`typing-extensions`/`tomli` guarded by `python_version<'3.11'`) maps to `%if %{python_version_nodots} < 311` — but the built flavours are all ≥311 so those branches never fire; keep them for correctness only.
- **Single-flavour *application* (a CLI/tool, not an importable library): drive the build off `%{primary_python}`, never a hardwired `python3`/`python313`.** A Python *application* builds for one flavour — the distribution's primary — and ships its entry-point binary, not a per-flavour module set. Use `%define pythons %{primary_python}` (tracks the primary as it moves, e.g. 3.13→3.14) instead of pinning `%define pythons python313`; reference paths via `%{$python_sitelib}` and deps via the `%{primary_python}-<dep>` form; do **not** use `%python_subpackages`/`%python_install_alternative`; ship the single `%{_bindir}/<tool>`. Even a `requires-python >= 3.13` floor is satisfied by `%{primary_python}` once the primary itself is ≥3.13 (`rpm --eval %{primary_python}` to check) — only hardwire a specific flavour in the rare case upstream needs a version the primary doesn't yet provide. (Real cases: skillspector, bugzilla-mcp.)
- **Renaming a package (`python-foo` → `foo`, or any flavour-change rename): carry the migration on the NEW package and target the *real old binary* name.** `osc copypac <oldprj> <oldpkg> <newprj> <newpkg>` (this preserves the `.changes` — keep it: rename the file content, prepend ONE rename entry, never start fresh; verify the separator count goes up by exactly one). Add `Provides: <old-binary> = %{version}-%{release}` + `Obsoletes: <old-binary> < %{version}` so `zypper dup` swaps it. **The Obsoletes/Provides target is the old *binary* RPM name, NOT the source name** — a singlespec module `python-foo` that set `%define pythons python313` builds `python313-foo`, so you Obsoletes `python313-foo` (not `python3-foo`); confirm via the old build's binary list (`osc api /build/openSUSE:Factory/standard/<arch>/<oldpkg>`). Keep that target a **literal** string — never macro-ize it (the historical name is fixed). Then submit the new package via its devel project and file a `osc deletereq openSUSE:Factory <oldpkg> -m 'renamed to <new>; superseded by sr#NNNN'`, cross-referencing the submit SR so reviewers see the rename pair. (Real case: python-bugzilla-mcp → bugzilla-mcp single-flavour app, Obsoletes literal `python313-bugzilla-mcp`.)
- **PEP 420 namespace packages (e.g. the `langgraph.*` family — langgraph + langgraph-checkpoint + langgraph-prebuilt all ship under `langgraph/` with no top-level `__init__.py`): every member must `%dir`-co-own the shared namespace dir.** In each `%files`, add `%dir %{python_sitelib}/<namespace>` (plus any shared intermediate dirs) and own only the subtree that package installs — never a sibling's subdir. Co-owning the parent `%dir` is correct and conflict-free; omitting it triggers `directories not owned by a package`. Verify with `rpm -qlp` that each owns its own subtree and only `%dir`s the shared parent.
- **`%{pypi_source}` does not always expand in the build env** (silently leaves a literal in `Source:`). For a stable, spec-cleaner-safe URL prefer the canonical redirect `https://files.pythonhosted.org/packages/source/<first-letter>/<dist_name>/<dist_name>-%{version}.tar.gz` (the `<dist_name>` is the PyPI *normalized* name, e.g. `langchain_core`, underscores not dashes).
- **Multi-flavour `pythonXX-<mod>` bindings inside a NON-python main package (a C/C++ lib that *also* ships the Python bindings for every flavour).** This is a different contract from a `python-<name>` module package — follow the `python-rpm-macros` README "`python_subpackage_only`" rules: (1) `%define python_subpackage_only 1` **before** `%python_subpackages`; (2) write the binding subpackage header with the **literal** `-n python-<modname>` prefix (e.g. `%package -n python-GDAL`), **never** `%{python_module <modname>}` — with ≥2 flavours the latter expands to a *space-separated list* and rpmbuild dies `error: Bad package specification: %package -n python311-GDAL python313-GDAL …`; (3) the files section is `%files %{python_files <modname>}` and inside it uses the **macro** form `%{python_sitearch}` / `%{python_sitelib}` — **NOT** `%{$python_sitearch}` (the `$`-form is only for shell inside `%python_expand`/`%build`/`%install`, and leaks literally into `%files` as `File must begin with "/": %{$python_sitearch}/…`); (4) inside the subpackage block `%{name}` is rewritten to `<flavor>-<modname>`, so require the **C library** by its literal name (`Requires: gdal = %{version}-%{release}`), not `%{name}`. `BuildRequires` still use `%{python_module …}` (list form is correct there). The primary flavour auto-`Provides: python3-<modname>`. (Real case: gdal 3.13.1 — five FTBFS iterations nailed exactly these four points.)
- **`%add_python <flavour>` (opt extra flavours beyond the single default `%pythons`) MUST come before the first `%{python_module …}` BuildRequires** — those BRs expand `%pythons` at parse time, so `%add_python` placed *after* them silently builds the BRs for the default flavour only (and rpm warns `Macro %pythons defined but not used within scope`). **spec-cleaner relocates `%add_python` down into the BR block AND rewrites it `%{add_python} <flavour>` (arg outside the braces → empty `$1` → broken)** — a documented semantic-exception: keep it unbraced near the top, reject spec-cleaner's move. (gdal `%add_python python311` → the python311/313/314 triplet.)
- **Per-flavour bindings against a single prebuilt C core (don't recompile the lib N×).** `find_package(Python)` picks the *newest* installed interpreter (e.g. 3.14), **not** `%primary_python` (3.13) — so a loop that "skips the primary" leaves the primary's `.so` unbuilt unless you force CMake onto it with **`-DPython_LOOKUP_VERSION=%{python3_version}`**. Then loop the *other* `%pythons` building only the bindings; map flavour→interpreter with `pybin="${python/python3/python3.}"` (python311→python3.11). Install each extra flavour with **`setup.py install_lib --skip-build --install-dir=%{buildroot}$sitearch`** + **`setup.py install_egg_info --install-dir=…`** — **NOT** `setup.py install … --install-scripts=<trash>`: with `--root` set, distutils `change_root` re-roots `--install-scripts` too, so the throwaway dir lands under the buildroot and fails the build as `installed (but unpackaged) … /usr/bin/<trash>/*`. Each flavour's extension is `_<mod>.cpython-3NN-*.so`; ship the noarch `/usr/bin/*.py` wrappers **once** in a separate subpackage. (gdal.)
