# Block 1 — Triage: does the package need updating?

## Checking for newer upstream versions

When asked "is this package out of date?" / "are there newer versions?", the honest answer needs the *current packaged version* (the spec's `Version:`) compared against the *latest real upstream release* — and several traps make a naive comparison wrong.

- **release-monitoring.org (Anitya) is behind the same Anubis anti-bot wall as the wiki.** `curl https://release-monitoring.org/api/v2/...` returns an HTML JS-challenge page ("Making sure you're not a bot!"), **not** JSON — even with a browser UA. So the usual one-stop version tracker is **not** scriptable here. Fall back to querying each package's actual upstream from its spec `URL:`/`Source:`.
- **GitHub/GitLab APIs are not gated** — use them. If `gh auth status` is logged in, use `gh api` (5000 req/hr) over anonymous curl (60 req/hr). GitLab: `https://gitlab.com/api/v4/projects/<owner>%2F<repo>/repository/tags`.
- **For a *bulk* "which of my packages need updating" sweep, use Repology** (`https://repology.org/api/v1/project/<name>`, not Anubis-gated). It does the version comparison itself: each entry has `repo`, `version`, and `status` (`newest`/`outdated`/`devel`/…). Read the `opensuse_tumbleweed` entry's status and compare to the `newest` entry. Set a real `User-Agent`, sleep ~0.4s between calls. **Three big caveats — always verify a Repology "outdated" hit before acting:**
  - **Repology tracks *published Tumbleweed*, which lags the devel project.** A package can show "outdated" in TW while its devel project already carries the new version. Confirm with the devel-project spec `Version:` (`osc develproject openSUSE:Factory <pkg>` → cat the devel spec), not just Repology. **Two sub-cases, very different actions:** (a) *update in flight* — devel is ahead and a Factory SR already exists/will land on its own → no action. (b) **stranded devel update — devel is ahead but the Factory forward SR was never filed, sometimes for *years*** → the fix is *not* packaging work, it's just **`osc sr <devel-project> <pkg> openSUSE:Factory`** to forward the already-built, already-green devel version. Check for a pending SR (`osc api '/search/request?match=action/target/@package="<pkg>" and action/target/@project="openSUSE:Factory" and state/@name="new"'`); if none and `osc results <devel> <pkg> -r openSUSE_Tumbleweed` is green, forward it. The `-> ?` Repology rows (where it can't resolve a "newest") are *especially* prone to this. (Real cases: **openmopac** — devel had 23.2.5 since 2026-05, Factory stuck at 23.1.2 → forwarded; **pspg** — devel actively maintained to 5.8.16, Factory frozen at **1.9.0 since 2019** → forwarded. Both were one `osc sr`, zero packaging.) (Now formalised as the Block-2 pre-flight HARD RULE — see `references/update-build.md` "Pre-flight: is this update already done or in flight?", mechanized by `scripts/preflight.sh`.)
  - **Multi-track upstreams can produce false "outdated" — but verify the track from the actual upstream layout before assuming it.** Some projects ship parallel stable/LTS and "advanced/devel" lines where a higher Repology `newest` belongs to a track you deliberately don't follow; a higher `newest` is not automatically the right target (same "verify by date / track, not by string" rule as below). **Counter-example / correction:** xmlrpc-c was long assumed to be such a case ("1.60.x super-stable vs 1.64.x advanced") — but the SourceForge file area shows **1.64.03 is in the *"Xmlrpc-c Super Stable"* folder**, i.e. it's the current super-stable release, the *same* line as 1.60.05 (just newer; upstream also dropped the `-c` from the `xmlrpc-1.64.03.tgz`/dir name). So it was genuinely outdated and a normal in-line update, not a track switch. Lesson: confirm the "track" by looking at where upstream actually files the release (SF folder, GitHub branch/release channel), don't trust a remembered assumption.
  - **Cross-project name collisions and format diffs**: Repology may map `xar`→"503", `sqlcipher`→a nonexistent "4.16.0" (different project), or compare `wireless-regdb` `20260530` vs `2026.05.30` (same release, different date format). Sanity-check implausible jumps. Repology project names are normalized, so many packages (python-* modules, the linphone stack, etc.) simply won't match by the openSUSE source name — "not found" ≠ "out of date".
- **A higher version *number* can be an *older* release — always verify by DATE, never by string/semver sort.** Projects renumber. The decisive check is the commit/tag date, not which string sorts higher:
  ```
  gh api "repos/<owner>/<repo>/commits/<tag>" --jq '.commit.committer.date'
  ```
  Real case: flowgrind's packaged `0.8.2` dates to **2021**, but tags `0.11`/`0.12` date to **2009** — upstream restarted its numbering (the 0.5→0.8.2 line is newer than the old 0.1→0.12 line). So `0.8.2` *is* the latest and "update to 0.12" would be a silent **downgrade to 12-year-old code**. Whenever a flagged "newer" version is a *lower* minor/major than the packaged one, or the jump looks implausible, pull the dates before acting.
- **`tags?per_page=1` returns git/alphabetical order, not chronological** — it is unreliable for "latest". Prefer `releases/latest` (the release the maintainer marked latest). `git_order` noise shows up as junk top tags like `test-tag-2022`, `io500-sc19`, `wholesym-v0.8.1`.
- **A tag without a *release object* usually has no upstream tarball.** `gh api repos/<o>/<r>/releases/tags/<tag>` 404 means only the GitHub auto-generated source archive (`/archive/refs/tags/<tag>.tar.gz`) exists — there is no maintainer-uploaded `.tar.bz2` or `.asc`. For an **autotools** project that archive lacks a pre-generated `./configure`, so adopting it forces a build-system change (run `autoreconf -fi` in `%prep`, add `autoconf`/`automake`/`libtool` BRs, drop the now-unverifiable `.asc`). That is a real cost — weigh it before "upgrading" to a tag the maintainer never released.
- **Git-snapshot packages (`0+gitYYYYMMDD`, `~<date>`, `+hg.<date>`) compare by date, not tag.** "Newer" means a newer upstream commit than the snapshot date; check `commits?per_page=1` HEAD date. Many such packages are deliberately ahead of the last upstream *tag* and are not out of date.
- **Some packages are *deliberately pinned* — do not "update" them.** Before proposing any version bump, read the latest `.changes` entries and spec comments for a pinning rationale (FIPS validation, an ABI/version a consumer hardcodes, a toolchain bootstrap). A package being years behind upstream is **not** automatically out of date. Real case: `boringssl` is pinned to `0.20210430` (tag `fips-20210429`) because Go 1.22/1.23 with `GOEXPERIMENT=boringcrypto` expects that exact FIPS module (boo#1236000) — bumping it would break Go's boringcrypto. When the pin is intentional, the "update" step is simply skipped (fix/clean only).
- **Report maintainership honestly.** `osc maintainer <project> <pkg>` shows package-level maintainers; when none are set it falls back to the **project** maintainers (the output then reads `maintainer of <project> :` without the `/<pkg>` suffix). Flag that distinction — "no dedicated maintainer, inherits project" changes who you should coordinate with before committing.
- **List a user's maintained packages in a project with the owner-search API:** `osc api '/search/owner?project=<prj>&filter=maintainer&user=<user>'` returns `<owner package=…/>` elements. Use it before working a devel project to know which packages are actually yours. An `<owner>` element with **no** `package=` attribute means the user is a **project-level** maintainer (inherits all packages in that project); elements **with** `package=` are explicit package-level maintainerships.
  - **`osc whois` first — the OBS username is not the local login.** The owner search's `user=` wants the OBS account name often differs from `$USER` and the email local-part. A wrong `user=` silently returns an empty `<collection/>`, which looks like "you maintain nothing" rather than an error. Confirm with `osc whois` before trusting an empty result.
  - **URL-encode `+` in project/package names as `%2B`.** A query against `devel:libraries:c_c++` written literally sends `c_c++`, and the server decodes `+`→space, yielding `devel:libraries:c_c` → `404 Project not found`. Use `devel:libraries:c_c%2B%2B`. (Same trap in any `osc api` query string, `/search/owner`, `/staging/...`, etc.)
- **`/search/owner?project=openSUSE:Factory&...` returns nothing for devel-level maintainers** — Factory maintainership is resolved through the *devel project*, and the owner index does not surface that as a Factory owner. To enumerate *all* packages a user maintains (across every project), query the person index instead:
  - Package-level: `osc api "/search/package?match=person/@userid='<user>'"` → every `<package project=… name=…/>` where the user is listed as a person. Filter out `home:*`, `openSUSE:Maintenance:*`, and `*:branches:*` noise to get the real working set.
  - Project-level: `osc api "/search/project?match=person/@userid='<user>'"` → projects where the user is a *project* maintainer (so they own every package there). (Grepping the XML for `name="…"` also matches `<repository name=…>`, so parse `<project>` elements, not raw `name=`.)
  The global `/search/owner?user=<user>&filter=maintainer` (no project) returns 0 owners — it is not a substitute. Map a devel package to its Factory presence with `osc develproject openSUSE:Factory <pkg>` (it returns `<devel-project>/<pkg>`; 404 = not in Factory).
